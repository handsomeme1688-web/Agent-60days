## git：

1.进入本地文件夹：cd path/to/filefolder
2.初始化git仓库: git init
3.添加文件到git仓库
    1.添加所有文件：git add .
    2.添加指定文件：git add filename
    3.添加指定文件夹：git add foldername/
4.提交文件到git仓库
    1.提交所有文件：git commit -m "提交信息"
    2.提交指定文件：git commit -m "提交信息" filename
    3.提交指定文件夹：git commit -m "提交信息" foldername/
    4.修补提交：git commit --amend -m "修补提交信息" --no-edit --reset-author（如果刚写完 commit 发现注释写错了，或者漏了一个小改动，可以用这个命令把改动并入上一次 commit，不产生新的记录。）
5.绑定 GitHub 远程仓库：git remote add origin https://github.com/你的用户名/你的仓库名.git
6.推送本地仓库到远程仓库：git push -u origin 分支名
    -u 参数的作用是把本地的 main 分支和远程的 main 分支关联起来，以后你只需要输入 git push 或 git pull 就能直接同步了。
7.从远程仓库拉取代码：
    1.同步最新代码并合并到本地分支：git pull，不合并到本地分支：git fetch 
    2.合并远程分支到本地分支：git merge origin/分支名
    3.本地为空，直接拉取远程代码：git clone https://github.com/用户名/仓库名.git

## Python：

### 1.函数

#### 默认参数：

新的power(x, n)函数定义没有问题，但是，旧的调用代码失败了，原因是我们增加了一个参数，导致旧的代码因为缺少一个参数而无法正常调用：

```py
power(5)

Traceback (most recent call last):
File "<stdin>", line 1, in <module>
TypeError: power() missing 1 required positional argument: 'n'
Python的错误信息很明确：调用函数power()缺少了一个位置参数n。
```

这个时候，默认参数就排上用场了。由于我们经常计算x2，所以，完全可以把第二个参数n的默认值设定为2：

```py
def power(x, n=2):
    s = 1
    while n > 0:
        n = n - 1
        s = s * x
    return s
```

这样，当我们调用power(5)时，相当于调用power(5, 2)：

```py
power(5)
25
power(5, 2)
25
```

而对于`n > 2`的其他情况，就必须明确地传入n，比如`power(5, 3)`。

从上面的例子可以看出，默认参数可以简化函数的调用。设置默认参数时，有几点要注意：

**一是必选参数在前，默认参数在后，否则Python的解释器会报错（思考一下为什么默认参数不能放在必选参数前面）；**

**二是如何设置默认参数: 当函数有多个参数时，把变化大的参数放前面，变化小的参数放后面。变化小的参数就可以作为默认参数。**

使用默认参数有什么好处？最大的好处是能降低调用函数的难度。

#### 注意！定义默认参数要牢记一点：默认参数必须指向不变对象！

Python 的默认参数只在函数“定义（编译）”时计算一次，而不是在“调用”时每次都计算。Python函数在定义的时候，默认参数`L`的值就被计算出来了，即`[]`，因为默认参数`L`也是一个变量，它指向对象`[]`，每次调用该函数，如果改变了`L`的内容，则下次调用时，默认参数的内容就变了，不再是函数定义时的`[]`了。

**Python中的可变数据类型：列表list、字典dict、集合set**

#### 可变参数（两种）：可变参数允许你传入0个或任意个参数，这些可变参数在函数调用时自动组装为一个tuple

#### 1.形参不变，调用可变

要定义出这个函数，我们必须确定输入的参数。由于参数个数不确定，我们首先想到可以把a，b，c……作为一个list或tuple传进来，这样，函数可以定义如下：

```py
def calc(numbers):
    sum = 0
    for n in numbers:
        sum = sum + n * n
    return sum
```

但是调用的时候，需要先组装出一个list或tuple：

```py
>>> calc([1, 2, 3])
14
>>> calc((1, 3, 5, 7))
84
```

#### 2.形参可变（参数前加 * ）

```py
def calc(*numbers):
    sum = 0
    for n in numbers:
        sum = sum + n * n
    return sum
```

```py
>>> calc(1, 2)
14
>>> nums = [1, 2, 3]
>>> calc(nums[0], nums[1], nums[2])
14
```

#### 关键字参数（参数前加 * *）：关键字参数允许你传入0个或任意个含参数名的参数，这些关键字参数在函数内部自动组装为一个dict

定义一个带有关键字参数的函数：

```py
def person(name, age, **kw):
    print('name:', name, 'age:', age, 'other:', kw)
```

函数`person`除了必选参数`name`和`age`外，还接受关键字参数`kw`。在调用该函数时，**可以只传入必选参数**：

```py
>>> person('Michael', 30)
name: Michael age: 30 other: {}
```

也可以**传入任意个数的关键字参数**：

```py
>>> person('Bob', 35, city='Beijing')
name: Bob age: 35 other: {'city': 'Beijing'}
>>> person('Adam', 45, gender='M', job='Engineer')
name: Adam age: 45 other: {'gender': 'M', 'job': 'Engineer'}
```

可以**把关键字参数先封装成一个字典**，再作为实参进行调用：

```py
>>> extra = {'city': 'Beijing', 'job': 'Engineer'}
>>> person('Jack', 24, **extra)
name: Jack age: 24 other: {'city': 'Beijing', 'job': 'Engineer'}
```

`**extra`表示把`extra`这个dict的所有key-value用关键字参数传入到函数的`**kw`参数，`kw`将获得一个dict，注意**`kw`获得的dict是`extra`的一份拷贝，对`kw`的改动不会影响到函数外的`extra`**。



### 参数解读

参数声明公式：                 **参数名 : 允许填入的类型 = 默认值**

用最简单的 `max_tokens: int | Omit | None = omit,` 来套用公式：

- **`max_tokens`**：参数名字。
- **`: int | Omit | None`**：冒号后面是**类型约束**。竖线 `|` 代表“或者”。也就是说，这个参数你可以填一个**整数（int）**，**或者**保持默认（Omit），**或者**填空（None）。
- **`= omit`**：等号后面是**默认值**。意思是如果你不传这个参数，系统会自动当成 `omit` 处理。

#### `1.(*, messages: ...)` 那个星号是什么鬼？

- **含义**：第一行开头的 `*` 是 Python 的一个特殊语法，叫做**命名关键字参数（Keyword-only Arguments）**。
- **怎么做**：它是在警告你，**后面的所有参数，在调用时必须写出参数名！**
  - ❌ 错误写法：`create(my_messages, "deepseek-v4-pro")`
  - 正确写法：`create(messages=my_messages, model="deepseek-v4-pro")`

#### 2. `messages: Iterable[ChatCompletionMessageParam]`

- **`Iterable[...]`**：意思是“可迭代的对象”，在绝大多数场景下，你直接理解成**列表（List）**即可。
- **`ChatCompletionMessageParam`**：这是 SDK 内部定义的一个 **Schema（数据图纸）** 的名字。它其实就是在规定你列表里的字典必须长成什么样（比如必须有 `role` 和 `content` 键）。
- **怎么做**：传一个我们之前学过的对话历史列表 `[{"role": "user", "content": "..."}]`。

#### 3. `model: ChatModel | str`

- **含义**：大模型的名字。它支持 `ChatModel`（SDK 预设的模型对象），**或者（|）** 一个普通的**字符串（str）**。
- **怎么做**：直接传字符串，比如 `model="deepseek-v4-pro"`。

#### 4. `Omit | None = omit` 频繁出现是什么意思？

在现代高阶 Python 库中，你会看到大量的 `| Omit | None = omit` 或 `| NotGiven = NOT_GIVEN`。

- **大白话**：这是厂商代码的“防呆设计”。用来区分“用户完全没传这个参数”**和**“用户故意传了一个 None”这两种情况。
- **怎么做**：**直接无视它**。这意味着这个参数是**可选的（Optional）**，你不写它完全没关系，系统有默认的底层处理。

#### 5. `logit_bias: Dict[str, int] | Omit | None = omit`

- **`Dict[str, int]`**：代表你需要传一个**字典（Dictionary）**，并且这个字典的键（Key）必须是**字符串（str）**，值（Value）必须是**整数（int）**。
- **怎么做**：如果需要用，就传类似 `{"token_id": 100}` 这样的 JSON 格式对象。

#### 6. `List[Literal['text', 'audio']]`

- **`Literal['text', 'audio']`**：字面量限制。意思是这个地方**只能填固定的死字**，要么填字符串 `'text'`，要么填字符串 `'audio'`，填别的单词都会报错。
- **`List[...]`**：外面套了 List，说明要用方括号包起来。
- **怎么做**：比如传 `['text']`。



### 2.IO编程

#### 2.1文件读写

#### 2.1.1读文件

要以读文件的模式打开一个文件对象，使用Python内置的`open()`函数，传入文件名和标示符：

```py
>>> f = open('/Users/michael/test.txt', 'r')
```

标示符**`'r'`表示读**，这样，我们就成功地打开了一个文件。

如果文件不存在，`open()`函数就会抛出一个`IOError`的错误，并且给出错误码和详细的信息告诉你文件不存在：

```py
>>> f=open('/Users/michael/notfound.txt', 'r')
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
FileNotFoundError: [Errno 2] No such file or directory: '/Users/michael/notfound.txt'
```

如果文件打开成功，接下来，调用`read()`方法可以**一次读取文件的全部内容**，Python把内容读到**内存**，用一个**`str`对象**表示：

```py
>>> f.read()
'Hello, world!'
```

最后一步是调用`close()`方法关闭文件。文件使用完毕后**必须关闭**，因为文件对象会占用操作系统的资源，并且操作系统同一时间能打开的文件数量也是有限的：

```py
>>> f.close()
```

由于文件读写时都有可能产生`IOError`，一旦出错，后面的`f.close()`就不会调用。所以，为了保证无论是否出错都能正确地关闭文件，我们可以使用`try ... finally`来实现：

```py
try:
    f = open('/path/to/file', 'r')
    print(f.read())
finally:
    if f: # 如果文件被打开了（文件对象f存在）
        f.close()
```

但是每次都这么写实在太繁琐，所以，Python引入了`with`语句来自动帮我们调用`close()`方法：

```py
with open('/path/to/file', 'r') as f:
    print(f.read())
```

这和前面的`try ... finally`是一样的，但是代码更佳简洁，并且不必调用`f.close()`方法。

调用`read()`会一次性读取文件的全部内容，如果文件有10G，内存就爆了，所以，要保险起见，可以反复调用`read(size)`方法，每次**最多读取size个字节的内容**。另外，调用`**readline()**`可以每次**读取一行**内容，调用`readlines()`**一次读取所有内容并按行返回`list`**。因此，要根据需要决定怎么调用。

如果文件很小，`read()`一次性读取最方便；如果不能确定文件大小，反复调用`read(size)`比较保险；如果是配置文件，调用`readlines()`最方便：

```py
for line in f.readlines(): # f.readlines()是一个列表
    print(line.strip()) # 把末尾的'\n'删掉
'''
strip() 是一个用来清理字符串“两端”空白字符的非常有用的方法。
当你从文件、网络或者用户输入中读取一行文本时，这行文本的末尾通常会自带一些看不见的符号，比如换行符（\n）、回车符（\r），或者行首行尾多余的空格和制表符（\t）。
strip() 的核心作用，就是把这些两端多余的“视觉垃圾”一刀切掉。
'''
```

#### file-like Object

像`open()`函数返回的这种有个`read()`方法的对象，在Python中统称为file-like Object。除了file外，还可以是内存的字节流，网络流，自定义流等等。file-like Object不要求从特定类继承，只要写个`read()`方法就行。

`StringIO`就是在内存中创建的file-like Object，常用作临时缓冲。

#### 二进制文件

前面讲的默认都是读取文本文件，并且是UTF-8编码的文本文件。要读取二进制文件，比如图片、视频等等，用`'rb'`模式打开文件即可：

```py
>>> f = open('/Users/michael/test.jpg', 'rb')
>>> f.read()
b'\xff\xd8\xff\xe1\x00\x18Exif\x00\x00...' # 十六进制表示的字节
```

#### 字符编码

要读取非UTF-8编码的文本文件，需要给`open()`函数传入`encoding`参数，例如，读取GBK编码的文件：

```py
>>> f = open('/Users/michael/gbk.txt', 'r', encoding='gbk')
>>> f.read()
'测试'
```

遇到有些编码不规范的文件，你可能会遇到`UnicodeDecodeError`，因为在文本文件中可能夹杂了一些非法编码的字符。遇到这种情况，`open()`函数还接收一个`errors`参数，表示如果遇到编码错误后如何处理。最简单的方式是直接忽略：

```py
>>> f = open('/Users/michael/gbk.txt', 'r', encoding='gbk', errors='ignore')
```

#### 2.1.2写文件

标识符`'w'`或者`'wb'`表示写文本文件或写二进制文件：

```py
>>> f = open('/Users/michael/test.txt', 'w')
>>> f.write('Hello, world!')
>>> f.close()
```

当我们写文件时，操作系统往往不会立刻把数据写入磁盘，而是放到内存缓存起来，空闲的时候再慢慢写入。只有调用`close()`方法时，操作系统才保证把没有写入的数据全部写入磁盘。所以，还是用`with`语句来得保险：

```py
with open('/Users/michael/test.txt', 'w') as f:
    f.write('Hello, world!')
```

要写入特定编码的文本文件，请给`open()`函数传入`encoding`参数，将字符串自动转换成指定编码。

以`'w'`模式写入文件时，如果文件已存在，会直接**覆盖**（相当于**删掉后新写入一个文件**）。如果我们希望追加到文件末尾怎么办？可以传入**`'a'`以追加**（append）模式写入。

#### 2.2 序列化(数据持久化，发生在内存中，不等同于把数据写入磁盘)

**把变量从内存中变成可存储或传输的过程称之为序列化**，在Python中叫pickling。**序列化之后，就可以把序列化后的内容写入磁盘**，或者通过网络传输到别的机器上。反过来，把**变量内容从序列化的对象重新读到内存里称之为反序列化**，即unpickling。Python提供了`pickle`模块来实现序列化。

首先，我们尝试把一个对象序列化并写入文件：

```py
>>> import pickle
>>> d = dict(name='Bob', age=20, score=88)
>>> pickle.dumps(d)
b'\x80\x03}q\x00(X\x03\x00\x00\x00ageq\x01K\x14X\x05\x00\x00\x00scoreq\x02KXX\x04\x00\x00\x00nameq\x03X\x03\x00\x00\x00Bobq\x04u.'
```

`pickle.dumps()`方法把任意对象序列化成一个`bytes`，然后，就可以把这个`bytes`写入文件。或者用另一个方法`pickle.dump()`直接把对象序列化后写入一个file-like Object：

```py
>>> f = open('dump.txt', 'wb')
>>> pickle.dump(d, f)
>>> f.close()
'''
file-like Object（类文件对象）其实就是你通过 open('dump.txt', 'wb') 创建出来的那个变量 f。

所谓的 File-like Object 是 Python 中的一个概念：它不一定是一个真正的硬盘文件，但它拥有像文件一样的方法（比如有 .write() 可以写入，有 .read() 可以读取）。

因为你在打开文件时使用的是 'wb' 模式（write binary，以二进制形式写入），pickle.dump(d, f) 已经把字典 d 变成二进制数据，直接写进了硬盘里的 dump.txt 文件中。
'''
# 写入完成后，重新打开文件进行读取
with open('dump.txt', 'rb') as f_read:
    content = f_read.read()
    print(content) 
    # 输出一串类似这样的二进制数据：b'\x80\x04\x95$\x00\x00\x00\x00\x00\x00\x00}\x94(\x8c\x04name\x94\x8c\x03Bob\x94\x8c\x03age\x94K\x14\x8c\x05score\x94KXXu.'
```

当我们要把对象从磁盘读到内存时，可以先把内容读到一个`bytes`，然后用`pickle.loads()`方法反序列化出对象，也可以直接用`pickle.load()`方法从一个`file-like Object`中直接反序列化出对象。我们打开另一个Python命令行来反序列化刚才保存的对象：

```py
>>> f = open('dump.txt', 'rb')
>>> d = pickle.load(f)
>>> f.close()
>>> d
{'age': 20, 'score': 88, 'name': 'Bob'}
```

#### 2.3 JSON——为了在不同的编程语言之间传递对象，就必须把对象序列化为标准格式

**json在python中只是一种格式（书写规范），其在python中是字符串。**

JSON表示的对象就是标准的JavaScript语言的对象，JSON和Python内置的数据类型对应如下：

| JSON类型   | Python类型 |
| ---------- | ---------- |
| {}         | dict       |
| []         | list       |
| "string"   | str        |
| 1234.56    | int或float |
| true/false | True/False |
| null       | None       |

Python内置的`json`模块提供了非常完善的Python对象到JSON格式的转换。我们先看看如何把Python对象变成一个JSON：

```py
>>> import json
>>> d = dict(name='Bob', age=20, score=88)
>>> json.dumps(d) # 注意这里是json调用dumps()方法，不是pickle
'{"age": 20, "score": 88, "name": "Bob"}'
print(type(json.dumps(d))) # <class 'str'>
```

`dumps()`方法返回一个`str`，内容就是标准的JSON。类似的，`dump()`方法可以直接把JSON写入一个`file-like Object`。

要把JSON反序列化为Python对象，用`loads()`或者对应的`load()`方法，前者把JSON的字符串反序列化，后者从`file-like Object`中读取字符串并反序列化：

```py
>>> json_str = '{"age": 20, "score": 88, "name": "Bob"}'
>>> json.loads(json_str) # 注意这里是json调用loads()方法，不是pickle
{'age': 20, 'score': 88, 'name': 'Bob'}
print(type(json.loads(json_str))) # <class 'dict'>
```

由于JSON标准规定JSON编码是UTF-8，所以我们总是能正确地在Python的`str`与JSON的字符串之间转换。

**`json.load(f)`**：就是“**以 JSON 格式去读**”。它不仅把文件里的文本读出来，还会自动把这堆文本“翻译并重组”成 Python 的列表、字典等真实对象。

**`json.dump(obj, f)`**：就是“**以 JSON 格式去写**”。它不仅把数据写入文件，还会自动在写入前把 Python 对象“翻译并拍扁”成符合 JSON 严格规范的字符串。

| **函数名**              | **包含 s (String)** | **处理的对象与流程**                                         | **它的工作原理（大白话）**                     |
| ----------------------- | ------------------- | ------------------------------------------------------------ | ---------------------------------------------- |
| **`json.dumps(obj)`**   | **有 `s`**          | 纯内存操作。把 Python 字典转换成 JSON 格式的**字符串**，并直接**返回**给一个**变量** | “把 Python 对象**变成** JSON 字符串”           |
| **`json.loads(s)`**     | **有 `s`**          | 纯内存操作。把一个 JSON 格式的**字符串**，反序列化还原成 Python 的**字典或列表** | “把 JSON 字符串**变成** Python 对象”           |
| **`json.dump(obj, f)`** | **没有 `s`**        | **涉及文件 `f`**， 把 Python 字典转换成 JSON 格式的字符串，但它**不返回**字符串给变量，而是直接**写入到一个打开的文件**里 | “把 Python 对象转成 JSON **并直接写入文件**”   |
| **`json.load(f)`**      | **没有 `s`**        | **涉及文件 `f`**，从一个打开的 **JSON 文件**中直接读取文本，并一步到位反序列化还原成 Python 对象 | “**直接读取文件**并把 JSON 还原成 Python 对象” |

**有 `s` 组 (`dumps` / `loads`)：** 只在**内存**里玩耍，负责【对象】和【字符串】之间的纯粹转换。不涉及磁盘 I/O。

**没 `s` 组 (`dump` / `load`)：** 必须拉上**磁盘文件**一起玩，顺手帮你把磁盘读写（I/O）的事情一起给办了。

#### 2.4  JSON进阶（https://docs.python.org/3/library/json.html#json.dumps）

```py
import json

class Student(object):
    def __init__(self, name, age, score):
        self.name = name
        self.age = age
        self.score = score

s = Student('Bob', 20, 88)
print(json.dumps(s))
# 上面的代码会报错。之所以无法把Student类实例序列化为JSON，是因为默认情况下，dumps()方法不知道如何将Student实例变为一个JSON的{}对象。
```

可选参数`default`就是**把任意一个对象变成一个可序列为JSON的对象（上面的表格里的那些，“可序列为JSON的对象”是Python中的基本数据类型的对象，而不是自定义的类对象）**，我们只需要为`Student`专门写一个转换函数，再把函数传进去即可：

```py
def student2dict(std):  # std是上面的定义的Student类的一个实例化对象
    return {
        'name': std.name,
        'age': std.age,
        'score': std.score
    }
'''
技术上，也可以转成列表：
def student2list(std):
    return [std.name, std.age, std.score] # 序列化后的 JSON 就会是：["Bob", 20, 88]
但是字典更清晰
'''
```

**字典的作用只是因为他刚好与json格式很像，所以临时转成字典，方便json格式匹配。**

这样，`Student`实例首先被`student2dict()`函数转换成`dict`，然后再被顺利序列化为JSON：

```py
>>> print(json.dumps(s, default=student2dict))
{"age": 20, "name": "Bob", "score": 88}
```

不过，下次如果遇到一个`Teacher`类的实例，照样无法序列化为JSON。我们可以偷个懒，把任意`class`的实例变为`dict`：

```py
print(json.dumps(s, default=lambda obj: obj.__dict__))
# 因为通常class的实例都有一个__dict__属性，它就是一个dict，用来存储实例变量。也有少数例外，比如定义了__slots__的class。
```

同样的道理，如果我们要把JSON反序列化为一个`Student`对象实例，`loads()`方法首先转换出一个`dict`对象，然后，我们传入的`object_hook`函数负责把`dict`转换为`Student`实例：

```py
def dict2student(d):
    return Student(d['name'], d['age'], d['score'])
```

```py
>>> json_str = '{"age": 20, "score": 88, "name": "Bob"}'
>>> print(json.loads(json_str, object_hook=dict2student))
<__main__.Student object at 0x10cd3c190>
```

## Python

### 1.装饰器（decorator）——在代码运行期间动态增加功能的方式

由于函数也是一个对象，而且函数对象可以被赋值给变量，所以，通过变量也能调用该函数。

```py
>>> def now():
...     print('2024-6-1')
...
>>> f = now
>>> f()
2024-6-1

```

函数对象有一个`__name__`属性（注意：是前后各两个下划线），可以拿到函数的名字：

```py
>>> now.__name__
'now'
>>> f.__name__
'now'

```

现在，假设我们要增强`now()`函数的功能，比如，在函数调用前后自动打印日志，但又不希望修改`now()`函数的定义，这种**在代码运行期间动态增加功能的方式，称之为“装饰器”**（Decorator）。

本质上，decorator就是一个返回函数的高阶函数。所以，我们要定义一个能打印日志的装饰器，可以定义如下：

```py
def log(func): # log 是“加工厂”的名字：它的作用是接收你的旧函数，并在厂房内部把 wrapper 生产出来。
    # 定义包装盒
    # wrapper 才是“包装盒”本身：它才是那个真正干活、加日志、调用原函数的外壳。
    def wrapper(*args, **kw):  # 万能参数,为了让 wrapper 这个外壳足够通用
        # 这里的 log 只是一个“空架子”,起作用的就是下面的一条print语句而已。这只是教学示例。
        print('call %s():' % func.__name__) # 这里的log的内容就是打印一下被@log注解的函数名。
        res = func(*args, **kw)
        # 当你加上 @log 后，你在代码里调用的函数，表面上是原函数，实际上就是在调用 wrapper。所以，wrapper 里面 return res，这个 res 就会直接送到你调用函数的地方。
        return res # 这里的 func 就是之前被藏在里面的原始 now 函数。它执行原始功能，打印出 2024-6-1。
    return wrapper # 把这个“包装盒”交付给你，让你以后能重复使用它
'''
为什么不能直接定义 wrapper，非要借 log 之手？
	wrapper 必须知道它要包装的“原函数”是谁。
	如果不用 log 把原函数（比如 now）当成参数传进来，wrapper 内部的 func(*args, kw) 就不知道该去调用谁了。
'''
```

观察上面的`log`，因为它是一个decorator，所以**接受一个函数作为参数**，并**返回一个函数**。我们要借助Python的@语法，把decorator置于函数的定义处：

```py
 @log
def now():
    print('2024-6-1')
```

调用`now()`函数，不仅会运行`now()`函数本身，还会在运行`now()`函数**前**打印一行日志：

```py
>>> now()
call now():
2024-6-1
```

把`@log`放到`now()`函数的定义处，相当于执行了语句：

```py
now = log(now)
```

由于`log()`是一个decorator，返回一个函数，所以，**原来的`now()`函数仍然存在**，只是现在**同名的`now`变量指向了新的函数**，于是调用`now()`将执行新函数，即在`log()`函数中返回的`wrapper()`函数。

```PY
'''
这句话的意思是：
	1.把原本的 now 函数塞进 log 这个“包装盒”里。
	2.包装盒加工完后，吐出了一个全新的、变漂亮了的函数（也就是 wrapper）。
	3.以后你再调用 now() 的时候，你喊的其实已经是那个“包装好的新函数wrapper”了。
但是，你原来的那个简陋的 now 消失了吗？没有，它只是被安全地藏在包装盒的里面呢。
'''
```

`wrapper()`函数的参数定义是`(*args, **kw)`，因此，`wrapper()`函数可以接受任意参数的调用。在`wrapper()`函数内，首先打印日志，再紧接着调用原始函数。

如果**decorator本身需要传入参数**，那就需要编写一个返回decorator的高阶函数，写出来会更复杂。比如，要自定义log的文本：

```py
def log(text):
    def decorator(func):
        def wrapper(*args, **kw):
            # C语言写法C用,连接，python用%连接。现在：print(f"{text} {func.__name__}():")
            print('%s %s():' % (text, func.__name__))
            # 为什么最内层能同时用 text 和 func？
            '''
            闭包（Closure）。
			最内层的 wrapper 就像是一个有记忆的魔法盒。虽然最外层的 log 和中间层的 decorator 执行完就退场了，但 wrapper 在出生的时候，已经把 text（'execute'）和 func（now）当成传家宝打包带走了。所以当你最终调用 now() 时，实际上是在执行 wrapper()：
	1.它翻开小账本，看到 text 是 'execute'，func.__name__ 是 'now'。
	2.于是快乐地打印出：execute now():
	3.接着调用原函数，打印出日期。
            '''
            return func(*args, **kw)
        return wrapper
    return decorator
'''
这三层套娃各自的“身份”是什么？

1.最外层 log(text) —— “总指挥 / 生产线定制员”
	它不是真正的装饰器，它是个定制工具。它的唯一任务，就是把你的自定义文本（比如 'execute'）接过来，然后为它量身定制一条生产线。
2.中间层 decorator(func) —— “真正的加工厂（也就是以前的 log）”
	这一层才是我们熟悉的那个加工厂。它负责把你的原函数（比如 now）接过来，准备开始包装。
3.最内层 wrapper(*args, kw) —— “最终的包装盒”
	真正干活的打工人。它既记住了最外层传进来的 text，又记住了中间层传进来的 func，最后组合在一起打印出来。
'''
```

这个3层嵌套的decorator用法如下：

```py
@log('execute') # 字符串execute就是上面的text
def now():
    print('2024-6-1')
    
'''
第一步：拆开最外层（执行 log('execute')）
Python 先看到 log('execute')，于是调用最外层的 log 函数。
log 把 'execute' 这个词死死记在心里。
走到最后一行 return decorator，log 把中间层的加工厂给吐了出来。
此时，原本的代码在后台等价于变成了： @加工厂。

第二步：拆开中间层（执行 decorator(now)）
现在变成了标准的装饰器流程。中间层的 decorator 接收了 now 函数。
它把 now 塞进了最内层的 wrapper 盒子里。
走到 return wrapper，它把组装好的成品包装盒吐了出来。

第三步：交货
最终，同名变量 now 接住了这个最内层的 wrapper。
'''    
>>> now()
execute now():
2024-6-1
```

和两层嵌套的decorator相比，3层嵌套的效果是这样的：

```py
>>> now = log('execute')(now)
```

我们来剖析上面的语句，首先执行`log('execute')`，返回的是`decorator`函数，再调用返回的函数，参数是`now`函数，返回值最终是`wrapper`函数。

以上两种decorator的定义都没有问题，但还差最后一步。因为我们讲了函数也是对象，它有`__name__`等属性，但你去看经过decorator装饰之后的函数，它们的`__name__`已经从原来的`'now'`变成了`'wrapper'`：

```py
>>> now.__name__
'wrapper'
```

因为返回的那个`wrapper()`函数名字就是`'wrapper'`，所以，需要**把原始函数的`__name__`等属性复制到`wrapper()`函数中**，否则，**有些依赖函数签名的代码执行就会出错**。

不需要编写`wrapper.__name__ = func.__name__`这样的代码，Python内置的`functools.wraps`就是干这个事的，所以，一个完整的decorator的写法如下：

```py
import functools

'''
@functools.wraps(func) 的作用：
在后台默默地把 func（原函数）的以下祖宗八代信息，全部强行复制给 wrapper：
	__name__（函数名）
	__doc__（函数的注释文档）
	__module__（函数属于哪个模块）
	__annotations__（函数的参数类型提示）
'''
def log(func):
    @functools.wraps(func)
    def wrapper(*args, **kw):
        print('call %s():' % func.__name__)
        return func(*args, **kw)
    return wrapper
```

或者针对带参数的decorator：

```py
import functools

def log(text):
    def decorator(func):
        @functools.wraps(func)
        # 只在wrapper上加装饰器是因为只有 wrapper 才是那个真正要“替代”原函数、要出去抛头露面的“替身”。
        def wrapper(*args, **kw):
            print('%s %s():' % (text, func.__name__))
            return func(*args, **kw)
        return wrapper
    return decorator
```

### 2.类型注解——**给变量、参数或返回值“贴标签”**，注明它应该是什么类型

Python 的类型注解**只是“提示”，没有“强制约束力”**。

#### 作用

1. 自动补全更智能
2. 代码即文档（不用再猜变量是什么）
3. 提早发现 Bug（静态代码检查）

#### 类型别名

类型别名是使用 [`type`](https://docs.python.org/zh-cn/3/reference/simple_stmts.html#type) 语句来定义的，它将创建一个 [`TypeAliasType`](https://docs.python.org/zh-cn/3/library/typing.html#typing.TypeAliasType) 的实例。 在这个示例中，`Vector` 和 `list[float]` 将被静态类型检查器等同处理:

```py
type Vector = list[float] # 给由浮点数组成的列表起了个外号，名为Vector
# 函数scale接收两个参数，一个是浮点数scalar，另一个是Vector类型的vector，vector其实是一个由浮点数组成的列表。
# 函数scale预计返回一个Vector类型的对象
def scale(scalar: float, vector: Vector) -> Vector:
    return [scalar * num for num in vector]

# 通过类型检查；浮点数列表是合格的 Vector。
new_vector = scale(2.0, [1.0, -4.2, 5.4])
```

类型别名适用于简化复杂的类型签名。例如:

```py
from collections.abc import Sequence

type ConnectionOptions = dict[str, str] # 一个字典，key和value都是字符串
type Address = tuple[str, int] # 元组类型，分别为字符串和整数
type Server = tuple[Address, ConnectionOptions] # 元组类型，分别为 Address 类型和 ConnectionOptions类型

def broadcast_message(message: str, servers: Sequence[Server]) -> None:
    ...

# 静态类型检查器会认为上面的类型签名
# 完全等价于下面这个写法。
def broadcast_message(
    message: str,
    servers: Sequence[tuple[tuple[str, int], dict[str, str]]]
) -> None:
    ...
```

[`type`](https://docs.python.org/zh-cn/3/reference/simple_stmts.html#type) 语句是在 Python 3.12 中新增加的。 为了向下兼容，类型别名也可以通过简单的赋值来创建:

```py
Vector = list[float]
```

或者用 [`TypeAlias`](https://docs.python.org/zh-cn/3/library/typing.html#typing.TypeAlias) 标记来显式说明这是一个类型别名，而非一般的变量赋值：

```py
from typing import TypeAlias

Vector: TypeAlias = list[float]
```

## Pydantic（用于解决类型注解没有强制约束力，有错也不报的问题）

**在 Pydantic 里，如果你想给数据制定一个‘规格说明书（模式）’，你就得自己写一个类，这个类必须继承 `BaseModel`，里面的属性必须写上类型注解。**

Pydantic 的行为总结为一句话：**在什么山头唱什么歌。数据源是什么，它就采用符合该数据源直觉的转换逻辑。**

**Pydantic 保证的是输出的类型和约束，而不是输入数据的类型和约束。**

字典里“验证”的意思是检查对错，但 Pydantic 做的比这多得多：

- **它会“强制类型转换（Coercion）”**：比如要求 `int`，输入 `"45"`，它不是单纯说“错”，而是把它变成 `45`。
- **它会“复制数据（Copying）”**：它会把输入的数据拷贝一份拿去改造，绝对不会污染或者修改你原本传进去的那条原始数据。

解析和验证分开：

- **解析（Parsing）**：专门指**把 JSON 字符串拆解成 Python 字典**的过程（比如把 `'{"a": 1}'` 变成 `{"a": 1}`）。
- **验证（Validation）**：专门指**把字典数据清洗、转换并塞进 `BaseModel` 变成模型实例对象**的整个过程。

### 基本模型使用

```py
from pydantic import BaseModel, ConfigDict
# BaseModel是Pydantic官方写好的类，我们自定义一个模型时直接继承他就行
class User(BaseModel): # 函数的括号里是参数，类的括号里就是继承的父类
    # User类包含的属性
	id: int
	name: str = 'Jane Doe' # 默认值
    
    # 所有字符串类型的字段，长度绝对不能超过 10 个字符。
	model_config = ConfigDict(str_max_length=10)  
```

在这个例子中，`User`是一个包含两个字段的模型：

- `id`，这是一个整数（使用[`int`](https://docs.python.org/3/library/functions.html#int)类型定义），并且是必需的。
- `name`，这是一个字符串（使用[`str`](https://docs.python.org/3/library/stdtypes.html#str)类型定义），不是必需的（它有一个默认值）。

然后即可实例化该模型：

```py
user = User(id='123')
```

在初始化这个对象的时候，Pydantic 会**自动自动执行所有的解析和数据校验**。如果没有抛出 `ValidationError`（数据校验错误）异常，就说明这个生成的模型实例是合法的、没问题的。

**隐藏细节：** 注意看，代码传入的是字符串 `'123'`。但因为 **Pydantic 知道 `id` 应该是个整数**，所以它在后台**自动帮你把字符串 `'123'` 转换（解析）成了数字 `123`**！

### 什么时候该继承 `BaseModel`？

你可以用这三个简单的问题来做决定：

| **问自己这个问题**           | **🟢 继承 BaseModel**                       | **❌ 声明普通类 class MyClass:**                |
| ---------------------------- | ------------------------------------------ | ---------------------------------------------- |
| **这个类的主要目的是什么？** | 纯粹用来**装数据**、定义结构               | 用来**写业务逻辑**、当成工具去干活             |
| **数据从哪里来？**           | 从网络、文件等**外部不受信任的渠道**传进来 | 自己代码内部创建，或者由可信的框架控制         |
| **我需要它的超能力吗？**     | 我需要自动校验、需要一键转成字典/JSON      | 我不需要校验，我只想老老实实写点面向对象的代码 |

模型的字段可以像对象的普通属性一样访问`user`：

```py
assert user.name == 'Jane Doe'  
assert user.id == 123  
assert isinstance(user.id, int)
```

这个模型实例可以使用 `model_dump()` 方法来进行**序列化**，序列化的结果为一个Python字典：

```py
assert user.model_dump() == {'id': 123, 'name': 'Jane Doe'}

# user.model_dump()，结果： 得到一个普通的 Python 字典 (dict)。
# json.dump() / json.dumps()，结果： 得到一个 JSON 格式的字符串 (str) 或者把数据写入 JSON 文件。
```

```py
1. 原始对象 (Pydantic Model)
    User(id=123, name='Jane Doe')
                 👇 
                 👇 使用 user.model_dump()
                 👇
 2. 半成品：Python 字典 (dict)
    {'id': 123, 'name': 'Jane Doe'}
                 👇
                 👇 使用 json.dumps()
                 👇
 3. 最终成品：JSON 字符串 (str)
    '{"id": 123, "name": "Jane Doe"}'
```

**1. `user.model_dump()` —— 负责“降维成字典”（不涉及到序列化）**

- **它干了啥：** 把复杂的 Pydantic 对象解构，把里面的数据提取出来，变成普通的 Python 字典。
- **为什么不算完全序列化：** 因为转换出来的字典依然是 **Python 独有的数据结构**。如果把这个字典直接通过网络发给一个 Java 或者 Go 语言写的程序，对方是根本无法直接识别的。它还没有走出 Python 的世界。

**2. `user.model_dump_json()` —— 负责“真正的序列化”**

- **它干了啥：** 一步到位，直接把 Pydantic 对象变成 **JSON 字符串**。
- **为什么它是真正的序列化：** 因为它输出的是纯文本字符串，已经彻底脱离了 Python 的束缚，可以跨平台、跨语言传输了。

### 模型方法和属性

**模型类**拥有以下方法和属性：

- [`model_validate()`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_validate)：根据 Pydantic 模型验证给定对象。
- [`model_validate_json()`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_validate_json)：根据 Pydantic 模型验证给定的 JSON 数据。
- [`model_construct()`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_construct)创建模型时不进行验证。
- [`model_dump()`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_dump)返回模型字段及其值的字典。
- [`model_dump_json()`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_dump_json)返回 JSON 字符串表示形式[`model_dump()`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_dump)。
- [`model_copy()`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_copy)返回模型的副本（默认情况下为浅拷贝）。
- [`model_json_schema()`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_json_schema)返回一个表示模型 JSON Schema 的 jsonable 字典。
- [`model_fields`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_fields)[`FieldInfo`](https://pydantic.dev/docs/validation/latest/api/pydantic/fields/#pydantic.fields.FieldInfo)：字段名称与其定义（实例）之间的映射。
- [`model_computed_fields`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_computed_fields)[`ComputedFieldInfo`](https://pydantic.dev/docs/validation/latest/api/pydantic/fields/#pydantic.fields.ComputedFieldInfo)：计算字段名称与其定义（实例）之间的映射。
- [`model_parametrized_name()`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_parametrized_name)：计算泛型类的参数化类名。
- [`model_post_init()`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_post_init)：在模型实例化并应用所有字段验证器后执行其他操作。
- [`model_rebuild()`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_rebuild)：重建模型模式，该模式也支持构建递归通用模型。

**模型实例**具有以下属性：

- [`model_extra`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_extra)：验证期间设置的额外字段。
- [`model_fields_set`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_fields_set)：模型初始化时显式提供的字段集。

### 自带数据转换

Pydantic可能会对输入数据进行类型转换，以强制其符合模型字段类型，在某些情况下，这可能会导致信息丢失。例如：

```py
from pydantic import BaseModel


class Model(BaseModel):
    # 通过 ConfigDict 全局开启严格模式
    model_config = ConfigDict(strict=True)
    
    # 局部开启
    # 1. 这个字段开启严格模式：必须传标准的 list
    items: list[int] = Field(strict=True)
    
    # 2. 这个字段没有开启：依然支持自动转换（比如把 3.000 变成整数 3）
    age: int
    
    a: int
    b: float
    c: str


print(Model(a=3.000, b='2.72', c=b'binary data').model_dump()) # 2.72被强制类型转换了
# 不过，Pydantic 提供了一种[严格模式]，在该模式下不进行任何数据转换。值必须与声明的字段类型相同。
#> {'a': 3, 'b': 2.72, 'c': 'binary data'}
```

集合也是如此。大多数情况下，你不应该使用抽象容器类，而应该使用具体的类型，例如[`list`](https://docs.python.org/3/glossary.html#term-list)：

```py
from pydantic import BaseModel


class Model(BaseModel):
	items: list[int]  # 官方推荐写法，使用具体的数据类型，例如list，而不应该使用抽象数据类型

# 数据结构的自动转换与规范化
print(Model(items=(1, 2, 3)))
#> items=[1, 2, 3]
```

**反例**，使用**抽象类**：

```py
from typing import Sequence
from pydantic import BaseModel

class MyModel(BaseModel):
    items: Sequence[int] # Sequence 是一个抽象概念，代表“序列”，列表和元组都算序列

    
'''
如果你传了一个元组 (1, 2, 3) 进去，Pydantic 会很困惑。因为元组本身就是一种 Sequence，它符合规则，所以 Pydantic 不会把它转换成列表，直接原样保留了。这就导致你后续拿到的 items 可能是列表，也可能是元组，数据格式不统一，后面写代码容易踩坑。
'''
```

### 多余数据的处理

默认情况下，Pydantic 模型**不会在您提供额外数据时报错**，这些值将被直接忽略：

```py
from pydantic import BaseModel


class Model(BaseModel):
    x: int


m = Model(x=1, y='a')
assert m.model_dump() == {'x': 1} # 多余数据不报错，直接丢弃
```

可以通过配置[`extra`](https://pydantic.dev/docs/validation/latest/api/pydantic/config/#pydantic.config.ConfigDict.extra)值来控制此行为：

```py
from pydantic import BaseModel, ConfigDict


class Model(BaseModel):
  x: int

  model_config = ConfigDict(extra='allow')


m = Model(x=1, y='a')  
assert m.model_dump() == {'x': 1, 'y': 'a'}
assert m.__pydantic_extra__ == {'y': 'a'}
```

该配置可以取三个值：

- `'ignore'`：默认情况下，如果你多传了数据，Pydantic **不会报错**，而是直接把多余的数据**无视掉（丢弃）**。
- `'forbid'`：不允许提供额外数据。**直接报错**！
- `'allow'`：允许提供额外数据，这些数据存储在`__pydantic_extra__`字典属性中。`__pydantic_extra__`可以显式地添加注解，以便对额外字段进行验证。

```py
class Model(BaseModel):
    x: int
    model_config = ConfigDict(extra='allow') # 👈 开启允许模式

m = Model(x=1, y='a')  

# 1. 导出时，多余的 y 也会被包含在字典里
assert m.model_dump() == {'x': 1, 'y': 'a'}

# 2. 多余的数据会被存在 __pydantic_extra__ 这个特殊的字典属性里
assert m.__pydantic_extra__ == {'y': 'a'}
```

### 嵌套模型

可以使用模型本身作为注解中的类型来定义更复杂的分层数据结构。

```py
from typing import Optional

from pydantic import BaseModel

# Foo 模型：基础组件，包含数字 count 和可选的浮点数 size（默认是 None）。
class Foo(BaseModel):
    count: int
    size: Optional[float] = None # Optional 不是关键字，它是由 Python 标准库 typing 模块提供的一个类型提示工具
    # Optional：管数据存不存在。Sequence：管数据是不是一个序列容器。

# Bar 模型：基础组件，包含带默认值的字符串 apple 和 banana。
class Bar(BaseModel):
    apple: str = 'x'
    banana: str = 'y'

'''
Spam 模型（核心）：它把上面两个模型当成了自己的数据类型！foo: Foo --> 代表 foo 这个字段必须符合 Foo 模型的结构。bars: list[Bar] --> 代表 bars 字段必须是一个由 Bar 模型组成的列表。
'''
class Spam(BaseModel):
    foo: Foo
    bars: list[Bar] # bars参数预计是一个由Bar类型的实例化对象组成的列表

# 参数写成字典是为了和前端发来的json格式匹配
# 在 Python 和 Pydantic 中，如果一个字段在定义时指定了默认值，或者被标记为可选的（Optional），那么在实例化它的时候，你完全可以不传这个参数，程序会自动使用默认值来补齐。
m = Spam(foo={'count': 4}, bars=[{'apple': 'x1'}, {'apple': 'x2'}])
print(m)
"""
Spam 类是一个大容器（车架），它虽然用 Foo 和 Bar（零件）来填充自己，但它拥有属于自己的结构定义和校验主权。在实例化时，Pydantic 会由内而外地先帮你把零件（子字典）实例化好，最后装入 Spam 交付给你。

foo=Foo(count=4, size=None) bars=[Bar(apple='x1', banana='y'), Bar(apple='x2', banana='y')] # 自动补齐其他参数

"""
print(m.model_dump())
"""
{
    'foo': {'count': 4, 'size': None},
    'bars': [{'apple': 'x1', 'banana': 'y'}, {'apple': 'x2', 'banana': 'y'}],
}
"""
```

Pydantic 还支持**自己套自己（自引用模型）**。

什么是自引用？最典型的例子就是**树状结构（比如文件夹结构）**：一个文件夹里可以包含文件，也可以**包含另一个文件夹**。

```py
from typing import Optional, List
from pydantic import BaseModel

class Node(BaseModel):
    name: str
    # 儿子节点还是 Node 本身（实现自己套自己）
    children: Optional[List['Node']] = None
```

### 重建模型结构——解决“先有鸡还是先有蛋” 的前向引用问题

当您在代码中定义模型类时，Pydantic 会分析类体，收集执行验证和序列化所需的各种信息，并将这些信息汇总到一个核心模式中。Pydantic 会评估模型的类型注解，以确定每个字段的有效类型。但是，**注解可能引用在创建模型类时尚未定义的符号**。为了解决这个问题，[`model_rebuild()`](https://pydantic.dev/docs/validation/latest/api/pydantic/base_model/#pydantic.BaseModel.model_rebuild)可以使用以下方法：

```py
from pydantic import BaseModel, PydanticUserError


class Foo(BaseModel):
    # 在 Python 中，为了不报错，会把还没定义的 Bar 用字符串包裹起来写成 'Bar'
	x: 'Bar'  # 此时 Python 和 Pydantic 都不知道 'Bar' 是个啥，因为它还没被定义
    
    
'''
如果你在定义完 Foo 后，立刻去调用 Foo.model_json_schema()（获取模型的 JSON 结构），Pydantic 就会崩溃并抛出错误（PydanticUserError），提示你：

“Foo 还没有定义完整；你应该先定义 Bar，然后调用 Foo.model_rebuild()。”
'''
try:
  Foo.model_json_schema()
except PydanticUserError as e:
  print(e)
  """
  `Foo` is not fully defined; you should define `Bar`, then call `Foo.model_rebuild()`.

  For further information visit https://errors.pydantic.dev/2/u/class-not-fully-defined
  """


class Bar(BaseModel):
  pass

# 为了解决这个问题，你需要在补全所有缺失的类之后，手动调用一次 model_rebuild()。
Foo.model_rebuild()
print(Foo.model_json_schema())
"""
{
  '$defs': {'Bar': {'properties': {}, 'title': 'Bar', 'type': 'object'}},
  'properties': {'x': {'$ref': '#/$defs/Bar'}},
  'required': ['x'],
  'title': 'Foo',
  'type': 'object',
}
"""
```

#### **文档中的正确操作顺序：**

1. 定义 `Foo`（里面用到了字符串形式的 `'Bar'`）。
2. **正式定义 `Bar` 类**。
3. 执行 `Foo.model_rebuild()` —— **告诉 Pydantic：“现在所有拼图都齐了，请重新解析并构建 `Foo` 的结构吧！”**
4. 此时再次调用 `Foo.model_json_schema()` 就能成功打印出完整的 JSON 结构了。

#### 什么时候需要主动用它？

虽然 Pydantic 通常很聪明，能自动检测什么时候需要重建，但在以下两种复杂场景下，你最好主动调用 `model_rebuild()`：

- **循环/递归模型（Recursive models）**：比如类 `Node` 里面又包含类 `Node`（树状结构）。

```py
from typing import List, Optional
from pydantic import BaseModel

class Node(BaseModel):
    name: str
    # 递归引用：Node 里面包含了一个由 Node 组成的列表（子节点）
    # 此时 Node 还没定义完，必须用字符串 'Node'。
    # 只有等整个缩进块结束，Python 才会正式创建这个类对象
    children: Optional[List['Node']] = None

# 定义完后，必须重建结构，否则 Pydantic 无法处理无限嵌套的校验
Node.model_rebuild()



# Python3.7新特性：
from __future__ import annotations #它会把整个文件里所有的类型提示自动隐式地变成字符串


# 如果加了这行声明，上面的代码就可以直接写成：
from __future__ import annotations
from typing import List, Optional
from pydantic import BaseModel

class Node(BaseModel):
    name: str
    children: Optional[List[Node]] = None  # 这里不加引号也不会报错了！

Node.model_rebuild()
```

- **泛型（Generics）**：涉及复杂的类型推导时。

```py
from typing import Generic, TypeVar
from pydantic import BaseModel


# 1.声明一个“类型变量”，名字叫 T（可以是任意大写字母，习惯用 T 代表 Type）
# T 现在就像是一个“变量名”，但它存的不是 1, 2, 3 这样的值，而是存“类型”（如 str, int, User）
T = TypeVar('T')


# 2.定义一个泛型模型，通过 Generic[T] 告诉 Pydantic：我这个类是一个模板，里面包含未知数 T
class Response(BaseModel, Generic[T]): # Generic[T]是类级别的形参列表，让Response类拥有被“填入类型”的能力
    status: str
    data: T  # 这里的 T 就是占位符。具体 data 是什么类型，取决于你用的时候往 T 里填什么。

    
# 当你结合前向引用使用泛型时，情况会变得极其复杂：
# 定义一个具体的业务数据模型 User
class User(BaseModel):
    name: str
    
    
#现在，我们要创建一个专门针对用户的响应结构。我们不需要重新写一个类，只需要像给函数传参一样，把 User 传给 Response：
user_response = Response[User]( # Response[User] 意味着：把模板里的 T 全部替换成 User 类。注意只能用[]传参
    status="success",
    data=User(name="张三") # 此时，data 属性的类型就被自动推导为了 User 类型
)


# 这种动态推导复杂类型时，往往需要手动 model_rebuild() 确保类型安全
Response[User].model_rebuild()
```

**为什么最后要调用 `model_rebuild()`？**

这其实是 Pydantic 内部的一种“安全保障机制”。

1. **动态生成新类**：当你写下 `Response[User]` 时，Pydantic 在后台实际上是通过**动态计算**临时为你创建了一个全新的隐藏类（比如叫 `Response_User`）。
2. **复杂的类型推导**：因为 `Response` 是动态组合 `User` 的，如果你的项目很大，`User` 类可能又引用了其他还没加载完的类（前向引用）。
3. **强制刷新**：`Response[User].model_rebuild()` 的目的，就是强制让 Pydantic 把这个刚刚动态组装出来的“新类”重新深度解析一遍，确保它内部所有的字段类型（不管嵌套多深）都已经百分之百安全、可用。

#### rebuild技术迭代

如果你对最外层的模型调用 `model_rebuild()`，它会**一口气把所有嵌套在里面的子模型、孙子模型的结构全部构建好**。这意味着，在执行这一步之前，**所有层级涉及到的类型都必须已经定义完毕**，否则就会报错。

### 验证数据

Pydantic 可以以**三种**不同的模式验证数据：*Python*、*JSON*和*字符串*。

#### 1.使用以下情况时会用到*Python*模式：

当你拿到的数据已经是 Python 的字典（`dict`）或对象时使用。

- **方法一：`__init__()` 构造函数**。也就是直接实例化：`User(id=123, name="James")`。必须用关键字传参。
- **方法二：`model_validate()`**。可以接收一个字典，或者另一个模型实例。
  - *代码示例：* `User.model_validate({'id': 123, 'name': 'James'})` 成功解析。

#### 2. JSON 模式（处理原始 JSON 数据）

当你从网络请求（API）收到原始的 JSON 字符串或字节流时使用。

- **方法：`model_validate_json()`**。
- **优势**：直接传入 JSON 字符串。这比你先用 `json.loads()` 把它转成字典，再传给 Pydantic 要**快得多**（内部有底层优化）。
  - *代码示例：* 传入 `{"id": 123, "name": 123}`，因为 `name` 应该是字符串却传了数字 `123`，直接触发 `ValidationError` 报错。

#### 3. 字符串模式（处理“全是字符串”的表单数据）

当你的数据来自网页表单、URL 参数或环境变量，导致**所有的数据类型都被变成了字符串**（比如数字 `123` 变成了 `'123'`）时使用。

- **方法：`model_validate_strings()`**。
- **优势**：它接收一个字典，但会强制用 **JSON 模式的逻辑** 去校验。这意味着它会非常聪明地把字符串 `'123'` 自动强制转换（Coerce）成整数 `123`，把字符串时间转换成标准的 `datetime` 对象。
  - *代码示例：* `{'id': '123', 'name': 'James'}` 传入后，`'123'` 被成功自动转成了整数 `123`。
  - *代码示例 2*：传入 `'signup_ts': '2024-04-01T12:00:00'`，成功转为了 Python 的 `datetime` 对象。

#### 总结：

Pydantic 极其强大的数据包容性：**给字典用 `model_validate()`，给 JSON 字符串用 `model_validate_json()`，给全是纯文本/字符串的表单数据用 `model_validate_strings()`。**

#### 重要对比与进阶特性

1. **更强的控制力**：相比于直接用 `User(...)` 实例化，使用 `model_validate_*()` 系列方法时，你可以额外传入很多控制参数，比如 `strict=True`（严格模式，不允许任何隐式类型转换，哪怕差个空格也会报错，如示例最后一部分所示）、`context`（校验上下文）等。
2. **行为差异（注意点）**：Python 模式和 JSON 模式在某些类型校验时的严格程度上会有微妙的差异。如果你拿到的不是 JSON，却想要 JSON 模式那样严格或特定的校验行为，官方建议：
   - 要么先用 `json.dumps()` 把数据转成 JSON 字符串，再用 `model_validate_json()`。
   - 要么直接使用 `model_validate_strings()`。

### 不验证强制创建模型

平时我们用 Pydantic 都是为了**数据校验**，但model_construct()这个方法允许你**完全跳过校验，直接“硬造”出一个模型对象**。

#### 三个主要的适用场景：

- **为了极致的性能**：如果你非常确定手头的数据是 100% 正确且合法的（比如刚从信任的数据库里读出来的数据），跳过校验可以省下 CPU 计算资源。
- **校验函数不是“幂等”的**：幂等（Idempotent）的意思是“运行一次和运行十次的结果一样”。如果你的校验函数运行多次会导致结果不一致，就不能重复触发校验。
- **校验函数有“副作用”**：比如你的校验函数里写了“每校验一次就往日志写一条记录”或“发送一个网络请求”，而你这次创建对象不想触发这些额外动作。

#### 警告！

> **“它不会做任何校验，意味着它能创造出完全残疾（非法）的模型。”**

比如你的模型要求 `id` 必须是整数，但你用 `model_construct(id="哈哈哈")` 传入一个乱七八糟的字符串，它**不会报错**，而是会照单全收，创建一个带病运行的对象。所以，**只有当你绝对信任数据源时才能用它！**

#### 技术提示

对于简单的模型，**正常校验甚至比跳过校验还要快**！ 如果你是为了性能想用它，请先做性能测试（Profile），不要盲目信任它会变快。

#### model_construct()方法底层行为特点

当你说“不进行任何校验”时，它到底有多彻底？文档列出了几条潜规则：

- **不会自动帮你嵌套转换**：如果类 `A` 里面包含类 `B`。平时你传入嵌套字典，Pydantic 会自动把内层字典转成类 `B` 的对象。但用 `model_construct()` 时它不管了，**你必须自己动手把内层字典转成对象**再传进去。
- **默认值依然有效**：如果你没传某个有默认值的字段，它还是会自动用默认值补齐。
- **连 `__init__` 构造函数都不执行**：哪怕你在类里自己写了一个 `def __init__(self):` 强行要在初始化时做点什么，不好意思，`model_construct()` 会直接绕过它，根本不调用它。

#### model_construct()方法处理多余数据（extra data）时

如果用户传了模型里根本没定义的字段（比如模型只有 `name`，但传了 `name` 和 `age`），它的处理逻辑取决于模型的 `extra` 配置：

- 如果配置是 `allow`（允许）：多余的 `age` 会乖乖存进内部的 `__pydantic_extra__` 字典里。
- 如果配置是 `ignore`（忽略）：多余的数据直接被丢弃。
- 如果配置是 `forbid`（禁止）：**重点来了！** 平时正常实例化时会直接抛出报错；但在这里，**它不会报错**，而是网开一面，仅仅是**把多余数据默默忽略掉**。

### 错误处理

核心机制：**一网打尽，绝不漏掉**

很多传统的校验库在遇到第一个错误时就会**抛出异常并停止运行**（比如发现 `list_of_ints` 有错就立刻崩溃）。但 Pydantic 不是。

Pydantic 的原则是：**无论数据里藏了多少个错误，它都会把整份数据全部检查完，最后只抛出一个统一的 `ValidationError` 异常。** 在这个异常里，它会像列清单一样，把所有翻车的字段、原因、错误类型和当时传进去的错误值**全部打包塞进去**，方便开发者**一次性看清所有问题**。

```py
from pydantic import BaseModel, ValidationError


class Model(BaseModel):

### 任意类实例

以前这个功能叫 `from_orm()`，现在叫 **`model_validate()`**。它可以让 Pydantic 直接读取一个普通 Python 对象（Object）的属性，只要属性名和 Pydantic 的字段名对得上，就能直接转换。

- **应用场景**：最常见的就是和 **ORM（对象关系映射，如 SQLAlchemy、SQLModel 等数据库框架）** 结合。你可以直接把数据库查询出来的结果丢给 Pydantic 做验证和过滤。
- **开启开关**：这个功能默认是关闭的。你必须**在 Pydantic 模型里配置 `model_config = ConfigDict(from_attributes=True)`**。默认情况下，Pydantic 只能验证字典（Dict）数据。加上这一行后，Pydantic 就获得了“超能力”，可以直接去读取一个 Python 对象的属性。

```py

from sqlalchemy import ARRAY, String
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column

from pydantic import BaseModel, ConfigDict, StringConstraints


# 1.数据库定义阶段
# Base: 这是 SQLAlchemy 2.0 的新写法，作为所有数据库模型的基类。
class Base(DeclarativeBase):
    pass

# 这是一个 ORM 模型（Object-Relational Mapping），用来映射数据库中的 companies 表。
class CompanyOrm(Base):
    __tablename__ = 'companies' # 数据库里的表名叫 companies
    
	# id: 整数，主键，不能为空。

    # public_key: 字符串，最大长度 20，建立了索引，不能为空，且全局唯一。
    public_key: Mapped[str] = mapped_column(
        String(20), index=True, nullable=False, unique=True
    )
    # domains: 字符串列表（数组），每个字符串最大长度 255。这是 PostgreSQL 等数据库支持的 ARRAY 类型。
    domains: Mapped[list[str]] = mapped_column(ARRAY(String(255)))

# 2. 数据模型定义阶段
class CompanyModel(BaseModel):
    
    # 核心开关：允许 Pydantic 从普通对象的属性中提取数据（比如从 co_orm.id 读取数据）
    # 默认情况下，Pydantic 只能验证字典（Dict）数据。加上这一行后，Pydantic 就获得了“超能力”，可以直接去读取一个 Python 对象的属性
    model_config = ConfigDict(from_attributes=True)

    id: int
    # 规定了 public_key 必须是字符串（str），限制字符串最大长度为 20
    public_key: Annotated[str, StringConstraints(max_length=20)]
    # 限制列表里的每一个字符串元素最大长度为 255
    domains: list[Annotated[str, StringConstraints(max_length=255)]]
     

# 3. 实例化与转换阶段

# 1. 模拟从数据库中查出了一条数据，创建了一个 ORM 对象
co_orm = CompanyOrm(
    id=123,
    public_key='foobar',
    domains=['example.com', 'foobar.com'],

# 打印它，结果是一个晦涩的内存对象地址

#> <__main__.CompanyOrm object at 0x0123456789ab>

# 2. 核心魔法：使用 model_validate 将 ORM 对象转换成 Pydantic 模型

# 打印转换后的模型，得到了干净、漂亮且经过验证的数据结构
print(co_model)

#### Field作用

在 Pydantic 中，**`Field` 是一个用来给字段“开小灶、加属性”的万能工具箱**。

当你写 `age: int` 时，你只是告诉了 Pydantic“这必须是个整数”。但如果你想对这个整数追加更多的限制（比如必须大于 0，或者设置一个默认值，或者在 API 文档里加个描述），普通的类型注解就无能为力了。这时就必须请出 `Field`。

它的核心作用主要有以下三点：

1. **数值与字符串约束（最常用）**
   - 对于数字：可以用 `ge`（大于等于）、`le`（小于等于）、`gt`（大于）、`lt`（小于）。
   - 对于字符串：可以用 `min_length`、`max_length`、`pattern`（正则匹配）。
2. **设置元数据（用于生成 API 文档）**
   - 你可以写 `Field(description="这是公司的官方联系电话")`，这样在使用 FastAPI 自动生成 Swagger 接口文档时，前端开发就能直接看到这行中文解释。
3. **别名与默认值**
   - 如果前端传过来的字段名叫 `phone-number`（带中划线，Python 不支持作为变量名），你可以用 `Field(alias="phone-number")` 来做完美映射。

#### r'.+@.+'的解读

1. #### 关于 `r''`：它是 Python 的“原始字符串”标识

**r''** 本身并不属于正则表达式的语法，它是 Python 语言的一种字符串前缀，叫做“**原始字符串**”（Raw String）。

- **为什么要加 `r`？** 在普通 Python 字符串中，**反斜杠 `\` 是用来表示转义字符的**（比如 `\n` 代表换行，`\t` 代表制表符）。 但在正则表达式里，我们也经常需要用到反斜杠（比如 `\.` 代表纯文本的点号）。
- **如果不加 `r`**：Python 会先尝试自己去解析反斜杠，这就**迫使你必须写成 `'\\.'`（用两个反斜杠来表示一个）**，代码会变得非常恶心。
- **加了 `r` 之后**：你就告诉了 Python：“**别动我的反斜杠，把它原封不动地传给正则表达式引擎！**”

> **结论**：`r''` 是为了**方便我们在 Python 里写正则表达式**而加的保护罩，它能保证正则表达式里的反斜杠不被 Python 误解。

2. #### 关于 `.+`：它其实是“两个”符号的组合

   你理解的“匹配任意字符”在结果上完全正确，但在原理上，**`.` 和 `\*` 应该切开来看**：

   - **`.`（点号）**：代表**匹配除换行符以外的“任意单个字符”**（无论是字母、数字、空格还是符号，只要是一个字就行）。
   - **`+`（加号）**：是一个数量修饰符，代表**前面的那个字符可以出现“1次或无数次”**。（*表示可以0、1、无数次）

   所以，它们组合在一起的完整逻辑是：

   - **`.`（任意字符） + `+`（出现任意多次） = `.+`（允许这里有任意长度的任何内容，至少有一个）。**

3. #### 使用正则表达式验证中国手机号码—— r'^1\d{10}$'

   - **`^`**：匹配字符串的**开头**。这意味着手机号必须从第一位开始匹配，前面不能有其他字符。
   - **`1`**：匹配数字 **1**。因为中国的手机号都是以 1 开头的。
   - **`\d`**：代表任意一个**数字**（等价于 `[0-9]`）。
   - **`{10}`**：表示前面的内容（即 `\d` 数字）**重复出现 10 次**。
   - **`$`**：匹配字符串的**结尾**。这意味着匹配到这里就必须结束，后面不能再有其他字符。
   - **`r'...'`**：在 Python 等语言中，开头的 `r` 表示这是一个“原始字符串”（Raw String），防止反斜杠 `\` 被当作转义字符处理。

#### 🚫 这 12 个符号必须转义：

| **符号**           | **在正则中的“魔法”含义**                 | **想匹配纯文本时怎么写** |
| ------------------ | ---------------------------------------- | ------------------------ |
| **`.`**            | 匹配除换行符外的任意单个字符             | `\.`                     |
| **`\`**            | 它是转义符号本身（所以它要转义自己！）   | `\\`                     |
| **`\*`**           | 匹配前面的字符 0 次或多次                | `\*`                     |
| **`+`**            | 匹配前面的字符 1 次或多次                | `\+`                     |
| **`?`**            | 匹配前面的字符 0 次或 1 次（非贪婪模式） | `\?`                     |
| **`^`**            | 匹配字符串的开头（或在 `[]` 内代表取反） | `\^`                     |
| **`$`**            | 匹配字符串的结尾                         | `\$`                     |
| **`(`** 和 **`)`** | 用于特征捕获组、分组                     | `\(` 和 `\)`             |

#### 💡 举几个生动的例子：

##### 例子 1：匹配加法算式 `"1+1=2"`

- ❌ 错误写法：`r'1+1=2'` ➡️ 正则会以为你想匹配“数字 1 出现一次以上，后面跟着 1=2”，这就会匹配失败。
- 正确写法：`r'1\+1=2'`

##### 例子 2：匹配一个官方价格 `"$99.9"`

- ❌ 错误写法：`r'$99.9'` ➡️ 在正则中 `$` 代表字符串结尾，它看不懂你想匹配美元符号。
- 正确写法：`r'\$99\.9'` （注意：`$` 和 `.` 都要转义）

##### 例子 3：匹配包含网址的括号 `"(baidu.com)"`

- ❌ 错误写法：`r'(baidu.com)'` ➡️ 正则会把括号当成“分组归类”，不当做文本。
- 正确写法：`r'\(baidu\.com\)'`



#### 懒人神技：如果你有一大串文本想直接匹配怎么办？

如果你需要匹配的字符串里包含了大量稀奇古怪的符号（比如 `r'a+b*c$d(e)[f]'`），手动一个个加反斜杠不仅容易漏，还很痛苦。

Python 的 `re` 模块自带了一个**一键“打回原形”\**的函数：\**`re.escape()`**。

```py
import re

raw_text = "a+b*c$d(e)[f]"
# 使用 re.escape 自动把所有需要转义的符号都加上反斜杠
safe_regex = re.escape(raw_text)

print(safe_regex)
# 输出: a\+b\*c\$d\(e\)\[f\]
```

## Python

### 1. 异步IO——解决CPU高速执行能力和IO设备的龟速严重不匹配的方法之一

多线程和多进程的模型虽然解决了并发问题，但是**系统不能无上限地增加线程**。由于**系统切换线程的开销也很大**，所以，一旦线程数量过多，CPU的时间就花在线程切换上了，真正运行代码的时间就少了，结果导致**性能严重下降**。

由于我们要解决的问题是**CPU高速执行能力和IO设备的龟速严重不匹配**，多线程和多进程只是解决这一问题的一种方法。

另一种解决IO问题的方法是异步IO。当代码需要执行一个耗时的IO操作时，它**只发出IO指令，并不等待IO结果**，然后就去执行其他代码了。一段时间后，**当IO返回结果时，再通知CPU进行处理**。

可以想象如果按普通顺序写出的代码实际上是没法完成异步IO的：

```py
do_some_code()
f = open('/path/to/file', 'r')
r = f.read() # <== 线程停在此处等待IO操作结果
# IO操作完成后线程才能继续执行:
do_some_code(r)
```

所以，**同步IO模型的代码是无法实现异步IO模型的**。

异步IO模型需要一个**消息循环**，在消息循环中，主线程不断地重复“**读取消息-处理消息**”这一过程：

```py
loop = get_event_loop() # 获取当前的“事件循环”对象。
while True:
    '''
    底层机制：
    	如果队列里有事件（比如用户点了鼠标、网络收到了一条数据、定时器到期了），它会立刻把这个事件取出来赋值给 event，然后程序继续往下走。
		如果队列里没有事件，这行代码通常会“阻塞”（暂停等待），让出 CPU 资源，直到有新的事件到来才被唤醒。
    '''
    event = loop.get_event() # 这是阻塞式（或轮询式）地从事件队列中获取下一个需要处理的“事件”。
    
    process_event(event) # 调用具体的处理函数，去执行这个事件对应的业务逻辑。
```

光看这四行代码本身，它确实是同步的——它在老老实实地、一步一步地循环。“异步”的精髓，并不在这四行代码本身，而是在于 event 是被谁放进来的，以及 process_event() 内部在做什么。

1. 关键在 `get_event()` 怎么“等”？（非阻塞等待）

如果这是同步模型，`get_event()` 在等待网络数据时，会把整个线程死死卡住，什么也干不了。

但在异步模型中，`get_event()` 底层借助了操作系统内核的功能（比如 Linux 的 `epoll`）。

- **它不是在等“某一个具体的网络请求完成”**。
- **它是在等“几千个网络请求中，有任意一个完成了就行”**。

如果暂时一个都没有，它就让出 CPU 睡大觉；一旦有任何风吹草动（比如某个请求的数据到了），它立刻醒来拿到这个 `event`。

2. 关键在 `process_event()` 里面执行了什么？这才是“异步”真正发生的地方。

假设你要从网络下载三个文件：文件 A（大）、文件 B（中）、文件 C（小）。

#### 传统的同步模型

你必须先写 `download(A)`，再写 `download(B)`。程序在下载 A 的时候，整个线程卡住，直到 A 下载完，才能去下载 B。



#### 真实的异步模型

1. **发出请求（不阻塞）**： 你同时发起了 A、B、C 的下载请求。发起请求的函数**立刻就返回了**，它没有等文件下载完，只是向操作系统注册了一句：“等下载完了，通知事件循环”。
2. **事件循环开始转动**： 因为发起请求不卡顿，代码迅速运行到了你的 `while True` 循环里。此时三个文件都在下载中。
3. **谁快谁先来（体现异步）**： 因为文件 C 最小，它先下载完了。操作系统把“C 下载完成”的事件丢进队列。
   - `loop.get_event()` 拿到了 C 的事件。
   - `process_event(C)` 开始执行，处理文件 C 的数据。
   - 处理完 C 后，文件 B 也下载完了，接着处理 B。
   - 最后，大文件 A 下载完了，处理 A。

为了更直观地理解，你可以参考以下事件循环的运行模型：

1. **事件产生（Event Sources）**：用户点击、网络请求、定时器触发等，这些都会产生“事件”。
2. **事件队列（Event Queue）**：所有产生的事件都会排队进入一个先进先出（FIFO）的队列中。
3. **事件循环（Event Loop）**：就是你提供的这段代码。它不断地从队列中取出事件（`get_event()`）。
4. **事件处理（Event Handler）**：将事件分发给对应的函数去执行（`process_event()`）。

消息模型是如何解决同步IO必须等待IO操作这一问题的呢？**当遇到IO操作时，代码只负责发出IO请求，不等待IO结果，然后直接结束本轮消息处理，进入下一轮消息处理过程。**当IO操作完成后，将收到一条“IO完成”的消息，处理该消息时就可以直接获取IO操作结果。

在“发出IO请求”到收到“IO完成”的这段时间里，**同步IO模型下，主线程只能挂起**，但**异步IO模型下，主线程并没有休息，而是在消息循环中继续处理其他消息。**这样，在异步IO模型下，一个线程就可以同时处理多个IO请求，并且没有切换线程的操作。对于大多数IO密集型的应用程序，使用异步IO将大大提升系统的多任务处理能力。



#### `yield` 

是 Python 中的一个关键字，它的作用是把一个普通的函数变成一个“生成器”（Generator）。`yield` 是一个“动作关键字”（运算符/控制流语句），它的作用对象是**整个“函数”和“执行流程”**，而不是某个具体的变量。

##### 1. 从最基本的代码来看 `yield`

使用 `return` 的普通函数

```py
def give_numbers():
    return 1
    return 2  # 这行代码永远不会被执行！因为 return 会直接结束并销毁函数
```

使用 `yield` 的生成器函数

```Python
def give_numbers_with_yield():
    print("开始做第一件事...")
    yield 1  # 返回 1，然后函数在这里“暂停”
    print("开始做第二件事...")
    yield 2  # 下次唤醒时，从这里继续，返回 2，再次“暂停”

# 使用它：
gen = give_numbers_with_yield() # 此时函数其实没有真正运行，只是得到了一个生成器对象

print(next(gen)) # 打印 "开始做第一件事..."，然后输出 1
print(next(gen)) # 打印 "开始做第二件事..."，然后输出 2
```

##### 2. 为什么要用 `yield`？它有什么不可替代的好处？

除了能用来做协程切换，`yield` 最经典的用途是**省内存（惰性求值）**。

假设你要处理 100 万个数据：

- 如果用 `return`，你必须在内存里开辟一个能装 100 万个元素的列表，一次性返回。如果数据量再大一点，你的电脑内存（RAM）直接就爆了。
- 如果用 `yield`，函数每次只产生**当前这一个**数据，传给外面用完后，再产生下一个。这意味着**无论数据量是 100 万还是 10 亿，它占用的内存永远只有当前处理的那一个数据的大小**。



#### **生成器（Generator）** 

就是 Python 中一种特殊的“按需生产数据的机器”。

##### **方法 A：**生成器函数（ `yield`）。

这是最常用的方式。只要函数里有 `yield`，调用它时就不会执行代码，而是返回一个生成器对象。

```py
def my_generator():
    print("开始生产...")
    yield "第一件商品"
    yield "第二件商品"

gen = my_generator() # 此时什么都没打印，只是得到了生成器
print(next(gen))    # 触发执行，打印 "开始生产..." 并输出 "第一件商品"
print(next(gen))    # 再次触发，输出 "第二件商品"
```

##### **方法 B：**生成器表达式。

把列表推导式的方括号 `[]` 改成圆括号 `()`，它就瞬间从“列表”变成“生成器”了。

```py
# 这是一个普通列表，10万个数字已经躺在内存里了
num_list = [x for x in range(100000)] 

# 这是一个生成器，内存里只有一行“生成规则”，不占空间
num_gen = (x for x in range(100000))
```



#### 总结yield和生成器

**yield的作用就是把函数变成一个生成器，而真正一次吐一条数据的是被yield改造过的函数，也就是那个生成器。**



#### 协程（coroutine，微线程）

和多线程比，协程有何**优势**？

1.最大的优势就是协程极高的执行效率。因为子程序切换不是线程切换，而是由程序自身控制，因此，**没有线程切换的开销**，和多线程比，线程数量越多，协程的性能优势就越明显。

2.第二大优势就是**不需要多线程的锁机制**，因为只有一个线程，也不存在同时写变量冲突，在协程中控制共享资源不加锁，只需要判断状态就好了，所以执行效率比多线程高很多。

因为协程是一个线程执行，那怎么利用多核CPU呢？最简单的方法是**多进程+协程**，既充分利用多核，又充分发挥协程的高效率，可获得极高的性能。

**Python对协程的支持是通过generator实现的。**

在generator中，我们不但可以通过`for`循环来迭代，还可以不断调用`next()`函数获取由`yield`语句返回的下一个值。

但是Python的**`yield`**不但**可以返回一个值**，它还可以**接收调用者发出的参数**。它会**把函数“暂停”在原地，等待下一次被唤醒**。

来看例子：

传统的生产者-消费者模型是一个线程写消息，一个线程取消息，通过锁机制控制队列和等待，但一不小心就可能死锁。

如果改用协程，生产者生产消息后，直接通过`yield`跳转到消费者开始执行，待消费者执行完毕后，切换回生产者继续生产，效率极高：

```py
# 代码的调度由Python虚拟机实现，与os无关！

'''
这里的r是不是相当于一个盘子，或者说库存、缓冲区。r被初始化为空，进入循环，yield r就是用于阻塞消费者自己来唤醒生产者，告诉生产者，我没东西用了，你快产，我等你生产好，并准备用n来接收生产者生产的东西。c.send(None)表示生产者在初始化时，不生产任何数据。接下来，生产者准备生产5个数据给消费者。 c.send(n)就是生产者阻塞自己来唤醒消费者，把生产的数据（这里的n是被简化的，直接被当作数据了。实际上可能是别的数据，而n只用来计数）使用send函数把数据n发送给消费者等待接收生产结果的n。        print('[CONSUMER] Consuming %s...' % n)就是相当于消费者在消费生产者送来的数据了，只不过这里简化为计数变量n了，因为生产者传过来的就是n。r = '200 OK'就是消费者通过改变了r的值，使自己进入下一轮，在运行yield r后，再次阻塞自己。r = '200 OK'这个值在消费者阻塞自己后，才返回给生产者告诉生产者我消费完了，你可以继续生产了，来唤醒生产者。生产者被唤醒后，执行它自己下面的代码，在这里是print('[PRODUCER] Consumer return: %s' % r)。然后这一轮结束。开始下一轮。
'''

# 消费者
def consumer():
    r = '' # 初始化返回结果为变空字符串。
    while True:
        n = yield r # 它的前半句（yield r）：把变量 r 的值返回给调用者，然后立即使函数暂停执行，像冬眠一样卡在这一行。
        # 它的后半句（n = ...）：当调用者下一次用 c.send(参数) 唤醒它时，传入的参数会被赋值给 n，然后程序解除冬眠，继续往下走。
        if not n: # 如果收到的 n 为空，退出循环，结束生成器。	
            return
        print('[CONSUMER] Consuming %s...' % n)
        r = '200 OK' # 处理完数据后，把 r 更新为状态码，准备在下一次循环的 yield 处返回给生产者。
# 生产者
def produce(c):
    c.send(None) # 这是启动生成器的固定语法（专业术语叫“预激” Prime）。consumer 函数开始运行，走到 n = yield r 时，把初始的 r（空字符串）返回，然后卡住暂停。此时控制权回到 produce。
    n = 0
    while n < 5: # 生产者开始循环，一共生产 5 个数据。
        n = n + 1
        print('[PRODUCER] Producing %s...' % n)
        r = c.send(n)
        print('[PRODUCER] Consumer return: %s' % r)
    c.close() # 5 次循环结束后，关闭消费者生成器。

c = consumer()
produce(c)
```

在 `n = yield r` 中：

- **右边的 `yield r` 是一整套动作**：把 `r` 吐出去 --> 让函数暂停 --> 等待 `send()` 传值回来。
- **左边的 `n =` 仅仅是承接结果**：当外面传值回来后，把这个值塞给变量 `n`。

**“为什么只把消费者做成生成器？是不是因为消费者慢？生产者能不能也是生成器？”**

答案是：这**与两个角色的“快慢”无关**，而是由单线程下“谁掌握主动权”的**控制流拓扑结构决定**的。 在单线程的协程世界里，生产者和消费者不能同时作为“被动响应”的生成器，必须有一个人站出来充当“主动驱动”的执行者（也就是大盲盒里的那个 `while` 循环推动者——生产者——作为主动驱动者）。否则，会陷入**双 yield 死锁**。如果是双yield,就需要引入了一个绝对权威的第三方裁判——**事件循环**（Event Loop）。下面的**async**就是这样解决这个问题的。

执行结果：

```py
[PRODUCER] Producing 1...
[CONSUMER] Consuming 1...
[PRODUCER] Consumer return: 200 OK
[PRODUCER] Producing 2...
[CONSUMER] Consuming 2...
[PRODUCER] Consumer return: 200 OK
[PRODUCER] Producing 3...
[CONSUMER] Consuming 3...
[PRODUCER] Consumer return: 200 OK
[PRODUCER] Producing 4...
[CONSUMER] Consuming 4...
[PRODUCER] Consumer return: 200 OK
[PRODUCER] Producing 5...
[CONSUMER] Consuming 5...
[PRODUCER] Consumer return: 200 OK
```

注意到`consumer`函数是一个`generator`，把一个`consumer`传入`produce`后：

1. 首先调用`c.send(None)`启动生成器；
2. 然后，一旦生产了东西，通过`c.send(n)`切换到`consumer`执行；
3. `consumer`通过`yield`拿到消息，处理，又通过`yield`把结果传回；
4. `produce`拿到`consumer`处理的结果，继续生产下一条消息；
5. `produce`决定不生产了，通过`c.close()`关闭`consumer`，整个过程结束。

整个流程无锁，由一个线程执行，`produce`和`consumer`协作完成任务，所以称为“协程”，而非线程的抢占式多任务。

最后套用Donald Knuth的一句话总结协程的特点：“**子程序就是协程的一种特例**。”



#### 使用asyncio（异步）

Python 3.5+：

- 把带有 `yield` 的生成器函数，改名叫 **`async def`**。
- 把里面负责拉手刹、交出控制权的 `yield`，改名叫 **`await`**。

**本质上，`async/await` 的底层就是“生成器 + yield”！**

用`asyncio`实现`Hello world`代码如下：

```py
import asyncio

async def hello():
    print("Hello world!")
    # 异步调用asyncio.sleep(1):
    await asyncio.sleep(1)
    print("Hello again!")

asyncio.run(hello())
```

`async`把一个函数变成`coroutine`类型，然后，我们就把这个`async`函数扔到`asyncio.run()`中执行。执行结果如下：

```plain
Hello world!
(等待约1秒)
Hello again!
```

`hello()`会首先打印出`Hello world!`，然后，`await`语法可以让我们方便地调用另一个`async`函数。由于`asyncio.sleep()`也是一个`async`函数，所以线程不会等待`asyncio.sleep()`，而是直接中断并执行下一个消息循环。当`asyncio.sleep()`返回时，就接着执行下一行语句。

把`asyncio.sleep(1)`看成是一个耗时1秒的IO操作，在此期间，主线程并未等待，而是去执行`EventLoop`中其他可以执行的`async`函数了，因此可以实现并发执行。

上述`hello()`还没有看出并发执行的特点，我们改写一下，让两个`hello()`同时并发执行：

```python
# 传入name参数:
async def hello(name):
    # 打印name和当前线程:
    print("Hello %s! (%s)" % (name, threading.current_thread))
    # 异步调用asyncio.sleep(1):
    await asyncio.sleep(1)
    print("Hello %s again! (%s)" % (name, threading.current_thread))
    return name
```

**用`asyncio.gather()`同时调度多个`async`函数**：

```python
# 事件循环是单车道，async def 只是声明"我可以让路"，await 才是真正让路的动作，而同步库压根不会让路。
async def main():
    # gather函数本身具有调度功能，并且会返回一个列表
    L = await asyncio.gather(hello("Bob"), hello("Alice")) #  gather 只是"把 10 个任务放上跑道"，await 才是发令枪。
    print(L)

asyncio.run(main()) 
```

执行结果如下：

```plain
Hello Bob! (<function current_thread at 0x10387d260>)
Hello Alice! (<function current_thread at 0x10387d260>)
(等待约1秒)
Hello Bob again! (<function current_thread at 0x10387d260>)
Hello Alice again! (<function current_thread at 0x10387d260>)
['Bob', 'Alice']
```

从结果可知，用`asyncio.run()`执行`async`函数，所有函数均由同一个线程执行。两个`hello()`是并发执行的，并且可以拿到`async`函数执行的结果（即`return`的返回值）。



**hello函数里面已经有await了，为什么gather函数前面也要加一个await?**

答：它们俩 `await` 的对象和目的完全不同。里面小任务的 `await` 是为了让小任务在遇到阻塞时让出 CPU；而`gather` 前面的 `await` 是为了让 **`gather` 所调用的所有函数彻底执行完，并拿到它们最终的返回值**。

如果不加外面的 `await`（错误示范）：

```py
async def main():
    # 仅仅是创建了并发任务，并没有等它们
    L = asyncio.gather(hello("Bob"), hello("Alice")) 
    print("经理下班了！") # 👈 这行代码会瞬间执行
    # 主程序在这里直接结束，Bob 和 Alice 还没睡醒就被销毁了，列表 L 里什么都没有
```



#### 企业级开发IO五大类

##### 1. 数据库 I/O（Database I/O）—— 最频繁的瓶颈

这是绝大多数商业系统中最核心、也是最慢的 I/O。每次你的代码需要读写数据，都要通过网络向数据库发起请求，并等待硬盘读写。

- **关系型数据库（CRUD）**：执行 SQL 语句查询或写入 MySQL、PostgreSQL、Oracle 等。例如：用户登录时查询用户信息，电商下单时扣减库存。
- **非关系型数据库/缓存**：读写 Redis、MongoDB 等。虽然 Redis 是内存数据库，速度极快，但“跨网络”的请求依然属于网络 I/O。
- **大数据量操作**：分库分表后的批量插入、复杂的表关联（JOIN）大表查询、数据库事务的提交（Commit）与回滚（Rollback）。

##### 2. 网络 I/O（Network I/O）—— 分布式系统的纽带

现代企业架构大多是分布式或微服务架构，微服务之间、系统与外网之间的所有通信，全部依赖网络 I/O。

- **微服务调用（RPC / HTTP）**：通过 Feign、gRPC、Dubbo 或者 RestTemplate 调用其他内部服务的接口。例如：订单服务调用支付服务、调用用户服务。
- **第三方 API 对接**：调用外部服务。如微信/支付宝支付回调、高德地图定位 API、云片网短信发送接口、微信公众号推送等。
- **Web 吞吐**：服务器接收前端（浏览器/App）发送的 HTTP 请求，并向前端返回 JSON 数据或 HTML 页面（即处理 Socket 连接）。

##### 3. 消息队列 I/O（Message Queue I/O）—— 异步解耦的关键

企业级开发为了应对高并发，经常使用消息队列进行“削峰填谷”。

- **消息发送（Produce）**：将订单消息、日志消息、通知消息推送到 Kafka、RabbitMQ、RocketMQ 等消息服务器。
- **消息消费（Consume）**：从队列中持续拉取（Pull）或接收（Push）消息并进行处理。这通常也是一个死循环的消息模型（正如你最开始看的那段代码）。

##### 4. 文件/磁盘 I/O（File / Disk I/O）—— 持久化的基石

直接对服务器本地硬盘或分布式文件系统进行读写。

- **日志写入（Logging）**：使用 Logback、Log4j2 等框架，在高并发下将系统日志、错误堆栈、审计日志写入到磁盘文件中。
- **文件上传与下载**：用户头像上传、Excel 报表批量导出、PDF 发票生成与下载。
- **分布式存储服务**：读写阿里云 OSS、腾讯云 COS、MinIO 等分布式对象存储系统（这其实结合了文件 I/O 和网络 I/O）。

##### 5. 系统与硬件 I/O（System & Hardware I/O）

一些特定的业务场景会涉及到更底层的硬件或内核级 I/O。

- **加密与安全（HTTPS/SSL）**：在建立网络连接时，CPU 与网卡配合进行 SSL/TLS 握手，调用系统的随机数生成器。
- **跨进程通信（IPC）**：在同一台服务器上，不同进程之间通过 Unix Domain Socket、管道（Pipe）或者共享内存传输数据。

##### 6. 示例

如果把`asyncio.sleep()`换成真正的IO操作，则多个并发的IO操作实际上可以由一个线程并发执行。

我们用`asyncio`的异步网络连接来获取sina、sohu和163的网站首页：

```python
import asyncio

async def wget(host):
    print(f"wget {host}...")
    # 第一步：打电话建立连接（网络 I/O），连接80端口（标准 HTTP 网页端口）:
    # 为什么要await?
    # 因为互联网是有延迟的，三次握手建立连接需要时间。踩中 await，这个下载员立刻对自己拉起手刹，交出控制权，让线程去瞅瞅别的人。
    
    

    reader, writer = await asyncio.open_connection(host, 80)
    # 第二步：把请求信件写好并寄出去，发送HTTP请求:
    # header 是标准的 HTTP 协议信件格式，翻译过来就是：“喂，我是浏览器，请把你们首页的数据发给我”。
    header = f"GET / HTTP/1.0\r\nHost: {host}\r\n\r\n"
    # GET /：我想**获取（GET）**你根目录（/，也就是首页）的网页文件。
    # HTTP/1.0：我使用的是 1.0 版本的网络协议。
    # Host: www.sina.com.cn：我知道你的名字叫新浪。
    
    
    # writer 只工作了 1 次：你就发了一句话（HTTP 请求头）。因为网络是全双工的    
    writer.write(header.encode("utf-8"))
    # 为什么要 await writer.drain()？
    # 因为write() 只是把信件放到了发送缓冲区，而 drain() 才是真正等待网卡把数据冲刷到网络线缆中。既然要等网卡干活，那就必须 await 挂起，把线程让出来。
    await writer.drain() # 等待数据真正冲出网卡

    # 第三步：一行一行读回信（循环网络 I/O），读取HTTP响应:
    while True: 
        # 为什么要 await？
        # 因为网络数据是一阵一阵传过来的（比如第一行到了，第二行还在路上）。每次调用 readline()，如果下一行数据还没完全进网卡，下载员就立马 await 挂起去睡觉，等数据到了被事件循环唤醒，打印出来，直到读完。
        # 在死循环里不停地读（读 N 次）
        line = await reader.readline() # 每次读一行网络数据
        if line == b"\r\n": # 读到空行说明网页头结束了，退出
            break
        print("%s header > %s" % (host, line.decode("utf-8").rstrip()))
    # 第四步：挂掉电话，彻底再见
    writer.close()
    await writer.wait_closed() # 等待操作系统彻底关闭这个网络通道
    print(f"Done {host}.")

async def main():
    # 调度总指挥：
    # 前面的 await 让经理（主程序）拉把椅子坐下，死等这三个网站全部抓取完毕才准下班。
    await asyncio.gather( # asyncio.gather(...) 把新浪、搜狐、网易三个下载任务同时推上了工位。
        wget("www.sina.com.cn"), 
        wget("www.sohu.com"), 
        wget("www.163.com")
    )

asyncio.run(main())
```

执行结果如下：

```plain
wget www.sohu.com...
wget www.sina.com.cn...
wget www.163.com...
(等待一段时间)
(打印出sohu的header)
www.sohu.com header > HTTP/1.1 200 OK
www.sohu.com header > Content-Type: text/html
...
(打印出sina的header)
www.sina.com.cn header > HTTP/1.1 200 OK
www.sina.com.cn header > Date: Wed, 20 May 2015 04:56:33 GMT
...
(打印出163的header)
www.163.com header > HTTP/1.0 302 Moved Temporarily
www.163.com header > Server: Cdn Cache Server V2.0
...
```

可见3个连接由一个线程并发执行3个`async`函数完成。

屏幕上的 `print` 之所以是交错的，只是因为它们都在公共的控制台屏幕（标准输出）上“抢镜头”汇报工作。但每个协程在内存里的数据是绝对隔离、互不干扰的。操作系统的 Socket 缓冲区**保证了数据进门时不混淆，**Python 虚拟机的独立栈帧**保证了处理数据时不串门，最后通过 **`return` 机制**保证了果实落地时各归各家。

##### **7. 小结**

`asyncio`提供了完善的异步IO支持，用`asyncio.run()`调度一个`coroutine`；

在一个`async`函数内部，通过`await`可以调用另一个`async`函数，这个调用看起来是串行执行的，但实际上是由`asyncio`内部的消息循环控制；

在一个`async`函数内部，通过`await asyncio.gather()`可以并发执行若干个`async`函数。



### 2. HTTP 

在 Web 开发中，每一个 URL 对应着网络上的一个“资源”（比如一个网页、一张图片、或者数据库里的一条用户数据）。而 `GET`、`POST`、`PUT` 等动词，就是你对这个资源下达的**指令**。

它们各自的含义和作用如下：

#### 1. `GET` —— 获取数据

- **代码**：`httpx.get(...)`
- **作用**：向服务器**索取**指定的资源。
- **场景**：比如你在浏览器输入网址看新闻、下载一张图片，或者在代码中获取某个接口的 JSON 数据。它只负责读取，不应该改变服务器上的数据。

#### 2. `POST` —— 提交/创建数据

- **代码**：`httpx.post(..., data={...})`
- **作用**：向服务器**提交**数据，通常用于让服务器**创建**新资源。
- **场景**：注册账号时提交表单、在论坛发帖、上传文件。你携带的 `data={'key': 'value'}` 就是要提交给服务器的内容。

#### 3. `PUT` —— 更新/替换数据

- **代码**：`httpx.put(..., data={...})`
- **作用**：向服务器发送数据，用以**完全替换**或更新已有的资源。
- **场景**：修改个人资料（把旧的资料全部覆盖掉）、修改文章内容。

#### 4. `DELETE` —— 删除数据

- **代码**：`httpx.delete(...)`
- **作用**：请求服务器**删除**指定的资源。
- **场景**：注销账号、删除一条朋友圈、删除购物车里的某件商品。

#### 5. `HEAD` —— 只要报头，不要内容

- **代码**：`httpx.head(...)`
- **作用**：和 `GET` 类似，但服务器**只返回 HTTP 响应头（Header）**，不返回具体的网页内容或数据实体（Body）。
- **场景**：当你只想检查一个大文件是否存在、文件有多大（Content-Length）、或者检查网页有没有更新，但又不想浪费流量下载整个文件时使用。

#### 6. `OPTIONS` —— 查看支持的操作

- **代码**：`httpx.options(...)`
- **作用**：询问服务器针对这个 URL **支持哪些 HTTP 请求方法**，或者检查安全权限（如跨域 CORS 预检）。
- **场景**：通常由浏览器或复杂的网络客户端在背后自动调用，用来探路。



### 3. HTTPX

**HTTPX** 是 Python 中一个下一代的、功能全面的 **HTTP 客户端**。

我们可以把 `httpx` 的用法分为两大阵营。

#### 两大阵营

1. 同步 API 阵营（和 `requests` 极其相似）

如果你不需要并发，或者是在写传统的单线程脚本，可以使用同步方法。

- **快捷方法：** `httpx.get()`, `httpx.post()`
- **推荐的客户端类：** `httpx.Client()`

```py
import httpx

# 方式 A：直接调用（每次都会新建连接，效率较低）
response = httpx.get("https://httpbin.org/get")

# 方式 B：使用 Client（推荐，支持连接池复用）
with httpx.Client() as client:
    response = client.get("https://httpbin.org/get")
    print(response.status_code)
```

2. 异步 API 阵营（专为 `asyncio` 设计）

当你在编写高并发程序、Web 爬虫或使用 FastAPI 等异步框架时，必须使用异步方法。

- **快捷方法：** `httpx.AsyncClient()`
- **核心特征：** 所有请求方法前必须加 **`await`** 关键字，且必须在 `async def` 函数内部使用。

```py
import httpx
import asyncio

async def fetch():
    # 必须使用 AsyncClient
    async with httpx.AsyncClient() as client:
        # 必须加 await！否则返回的是一个协程对象，而不是响应结果
        response = await client.get("https://httpbin.org/get")
        print(response.status_code)

# 启动异步任务
asyncio.run(fetch())
```

#### 回复内容

HTTPX 会自动将响应内容解码为 Unicode （万国码/统一码）文本。

2. Unicode 和 UTF-8 是什么关系？

很多人经常把 Unicode 和 UTF-8 搞混。这里有一个非常形象的比喻：

- **Unicode 是“字典/标准”**：它只负责给字符定编号（比如：“中”字等于数字 20013）。但它没有规定这个数字在电脑里具体怎么存成 `0` 和 `1`。
- **UTF-8 是“具体的实现方式/编码规则”**：它是为了节约空间而设计的一种把 Unicode 编号转换成二进制字节的方法。

```py
>>> r = httpx.get('https://www.example.org/')

```

您可以查看将使用哪种编码来解码响应。

>>> r.encoding # 查看使用的是哪种编码规则
'UTF-8'
```

在某些情况下，响应可能不包含显式编码，在这种情况下，**HTTPX 将尝试自动确定要使用的编码**。

```py
>>> r.encoding

>>> r.text
'<!doctype html>\n<html>\n<head>\n<title>Example Domain</title>...'
```

```



#### 二元响应内容

当你要下载的东西**不是普通的文字**（比如一张 `.jpg` 图片），你就不能用 `r.text` 了，而是要用 `r.content` 来获取它的**原始二进制字节（Binary Bytes）**。

```py
>>> r.content
b'<!doctype html>\n<html>\n<head>\n<title>Example Domain</title>...'
# 这里的 b'...' 开头的 b 代表 Bytes（字节）。计算机里的一切文件（图片、视频、PDF）本质上都是一串由 0x00 到 0xFF 组成的十六进制字节流。
```

’ 任何`gzip`HTTP`deflate`响应编码都将被自动解码。如果`brotlipy`已安装，则`brotli`支持响应编码。如果`zstandard`已安装，则`zstd` 也支持响应编码。‘

**解读**：这里的“编码”不是前面说的文字编码（如 UTF-8），而是指**数据压缩算法**。

**背后的故事**：为了让你打开网页更快、省流量，服务器在发货前，通常会用 `gzip` 或 `brotli` 把数据“压缩成一个拉链包”发给你。

**HTTPX 的便利之处**：HTTPX 收到这个压缩包后，**会自动在后台帮你“解压”**。你不需要自己写代码去解压 gzip，直接调用 `r.content` 拿到的就是解压后、现成可用的原始字节。

例如，要从请求返回的二进制数据创建图像，可以使用以下代码：

```py
# 实例演示：把二进制字节变成一张真正的图片
>>> from PIL import Image
>>> from io import BytesIO
>>> i = Image.open(BytesIO(r.content))
```

这是一个非常经典的把网络图片“无盘下载”到内存中处理的操作。

- **`r.content`**：从网络上下载下来的图片的**原始二进制数据**（一堆 `0101` 电脑管它叫字节，但现在它还不是程序能看懂的“图片格式”）。
- **`BytesIO(r.content)`**：这步非常巧妙。Python 的图片库（PIL）通常只能从硬盘上的文件（比如 `D:/meinv.jpg`）里读取图片。`BytesIO` 的作用是**在内存中模拟一个虚拟文件**，把那堆二进制字节伪装成一个“文件对象”，这样就不用先把图片保存到硬盘，省去了读写硬盘的开销。
- **`Image.open(...)`**：PIL 图像库打开这个虚拟文件，把它真正识别并解码为一张**图像对象 `i`**。接下来你就可以对这张图片进行裁剪、滤镜或者直接保存到本地了。



**总结速查：`r.text` vs `r.content`**

| **想要获取的内容**   | **应该使用谁** | **返回的数据类型**     | **典型应用场景**               |
| -------------------- | -------------- | ---------------------- | ------------------------------ |
| **纯文本/网页源码**  | `r.text`       | Python 字符串 (`str`)  | 爬取小说、新闻、获取 JSON 数据 |
| **多媒体/文件/图片** | `r.content`    | 二进制字节流 (`bytes`) | 下载图片、下载 MP3、下载 PDF   |



#### JSON 响应内容

 **HTTPX 如何完美支持和处理 Web API 中最常见的 JSON 数据格式**？——Web API 的响应通常会被编码为 JSON 格式。

```py
>>> r = httpx.get('https://api.github.com/events')
>>> r.json()
[{u'repository': {u'open_issues': 0, u'url': 'https://github.com/...' ...  }}]
```

- **背后的痛点**：如果你用 `r.text` 去拿 API 数据，你拿到的只是一个“大字符串”。如果你想提取里面的 `name` 或 `age`，你必须自己写代码去解析这个字符串，非常麻烦。
- **HTTPX 的解法**：当你调用 `r.json()` 时，HTTPX 会自动做两件事：
  1. 自动把这个长长的 JSON 文本解析开。
  2. 将其转换成**标准的 Python 数据类型**（通常是 Python 字典 `dict` 或嵌套了字典的列表 `list`）。

一旦转换完成，你就可以直接像操作普通 Python 变量一样，用键值对去提取数据了：

```py
data = r.json()
print(data[0]['repository']['url'])  # 优雅地直接层层提取数据
```

文档里的 `u'repository'` 是什么意思？

你可能注意到了文档输出里有很多小写的 `u`，比如 `u'repository'`。

- **科普一下**：这是旧版本 Python（Python 2）遗留下来的痕迹，`u` 代表 **Unicode 字符串**。
- **现代情况**：现在的 Python 3 中，所有字符串默认都是 Unicode 了，所以你在现代环境里运行 `r.json()`，不会再看到这些丑陋的 `u` 了，直接就是干净的 `{'repository': ...}`。



#### 自定义标题(请求头)

**请求头** 就像是你发信时在信封上贴的“额外标签”或“备注信息”。它不包含真正的请求内容，但它会告诉服务器**关于你（客户端）的元信息**，比如你用的是什么浏览器、你支持接收什么格式的数据等。当你需要向服务器证明身份（防反爬）、传递登录令牌（Token）或进行特殊安全验证时，都可以通过 `headers` 参数传入一个 Python 字典，从而优雅地控制你的请求特征。

要在传出请求中包含其他标头，请使用`headers`关键字参数：

```py
>>> url = 'https://httpbin.org/headers'

'''
User-Agent（简称 UA）是 HTTP 协议中最常用的一个标头，它的作用是表明身份。当你用 Chrome 浏览器上网时，Chrome 会自动发送一串包含 “Chrome/120.0...” 的字符串，告诉服务器：“我是谷歌浏览器”。

为什么要自定义它 ? ：
	默认情况：如果你直接使用 httpx.get(url)，HTTPX 默认发过去的 UA 是类似 python-httpx/0.27.0。	
	反爬虫对抗：很多网站看到 python-httpx 或 python-requests 这样的 UA，就知道这是个爬虫程序，而不是真正的正常人类用户，于是会直接拒绝访问（返回 403 报错）。
	伪装与标示：通过伪装这个标头，你可以把自己伪装成真正的浏览器（比如 Chrome）；或者像文档中一样，将其改成 my-app/0.0.1，礼貌地告诉服务器：“这是我自己写的某个特定 App 的客户端”。
'''
>>> headers = {'user-agent': 'my-app/0.0.1'}



# 当你把包装好的字典 headers 传给 headers 关键字参数后，HTTPX 就会在发送网络请求时，把这些自定义的信息打包进“信封”里一起发给服务器。
>>> r = httpx.get(url, headers=headers)
```

**常见的高级玩法的其他“标头”**

除了 `user-agent` 之外，我们在实际开发中还经常自定义以下几种标头（同样是用字典传进去）：

- **`Authorization`（授权认证）**：用来放 Token 或密码。很多高级 API 需要你登录后才能访问，你就需要把你的“身份令牌”放在这里发过去。

  Python

  ```py
  headers = {'Authorization': 'Bearer your_secret_token'}
  ```

- **`Referer`（来源页）**：告诉服务器你从哪个网页链接跳过来的。有些防盗链的图片网站，会检查 `Referer` 是不是自己公司的域名，如果不是就不给显示。

- **`Cookie`（小饼干）**：用来维持登录状态。



#### 发送表单编码数据

当你在网页上填写表单（比如输入账号密码、勾选兴趣爱好），然后点击“提交”按钮时，浏览器在幕后发送的就是这种 **表单编码数据（Form-encoded data）**。

某些类型的 HTTP 请求，例如POST`POST`和`PUT`GET 请求，可以在请求体中包含数据。一种常见的包含方式是使用表单编码数据，这种方式常用于 HTML 表单。

```py
>>> data = {'key1': 'value1', 'key2': 'value2'}
>>> r = httpx.post("https://httpbin.org/post", data=data)
>>> print(r.text)
{
  ...
  "form": {
    "key2": "value2",
    "key1": "value1"
  },
  ...
}
```

**工作原理**：当你使用 `httpx.post` 并传入 `data` 参数时，HTTPX 会自动把这个字典转换成 `key1=value1&key2=value2` 格式的字符串。

**关键区别**：**它不放在网址（URL）里！** 它是被打包藏在 HTTP 请求的“主体（Body）”中发送的。

**返回结果**：通过打印 `r.text` 可以看到服务器收到的内容，服务器成功在 `"form"` 字段里解析出了你提交的键值对。

表单编码数据还可以包含给定键的多个值。

```py
>>> data = {'key1': ['value1', 'value2']}
>>> r = httpx.post("https://httpbin.org/post", data=data)
>>> print(r.text)
{
  ...
  "form": {
    "key1": [
      "value1",
      "value2"
    ]
  },
  ...
}
```



#### `params` vs `data` 到底用哪个？

这是很多初学者最容易混淆的地方。因为它们都可以接收字典，而且支持列表，但它们的底层行为完全不同！

- **`params`（查询参数）**：
  - **位置**：附加在 **URL 网址** 后面（可见）。
  - **作用**：通常用于 `GET` 请求，用来**筛选、过滤**数据（如：`?page=2&size=10`）。
  - **数据量**：因为网址长度有限制，不能传太大的数据。
- **`data`（表单数据）**：
  - **位置**：塞在 **HTTP 请求体（Body）** 里面（URL 界面上看不到）。
  - **作用**：通常用于 `POST` 或 `PUT` 请求，用来**提交、创建**数据（如：账号密码、修改内容）。
  - **数据量**：没有严格的长度限制，可以传很大的文本或表单。



#### 发送多部分文件上传

当我们在网页上点击“上传头像”、“上传 PDF 报告”按钮时，浏览器在幕后使用的就是这种 **多部分表单编码（Multipart Encoding）** 技术。它允许在同一个 HTTP 请求中，**既传输普通的文本，又传输复杂的二进制文件**。

您还可以使用 HTTP 多部分编码上传文件：

```py
>>> with open('report.xls', 'rb') as report_file:
...     files = {'upload-file': report_file}
...     r = httpx.post("https://httpbin.org/post", files=files)
>>> print(r.text)
{
  ...
  "files": {
    "upload-file": "<... binary content ...>"
  },
  ...
}
```

**`'rb'` 模式**：打开文件时必须使用 `'rb'`（读取二进制），因为文件是以原始字节流的形式发送的，不能用普通文本模式打开。

**`files=files`**：这里不能用 `data`，**必须把文件字典传给 `files` 关键字**。HTTPX 看到后，会自动把请求头改成 `multipart/form-data`，并以**切块的形式**把文件发送出去。

**返回结果**：服务器收到后，文件数据会存放在结果的 **`"files"`** 字段中。

您还可以通过使用元组作为文件值来显式设置文件名和内容类型。有时候，你的文件是从内存里生成的，或者你想在发送时改个名字，或者某些严格的服务器需要检查文件的**具体类型（MIME Type）**。：

```py
>>> with open('report.xls', 'rb') report_file:
...     files = {'upload-file': ('report.xls', report_file, 'application/vnd.ms-excel')}
...     r = httpx.post("https://httpbin.org/post", files=files)
>>> print(r.text)
{
  ...
  "files": {
    "upload-file": "<... binary content ...>"
  },
  ...
}
```

这时候，字典的值不再是简单的文件对象，而是一个**元组（Tuple）**，格式为： `('文件名', 文件对象, '文件类型/MIME Type')`

- **`application/vnd.ms-excel`**：这就是 Excel 文件的标准 MIME 类型（就像网页的 `text/html`，图片的 `image/jpeg`）。显式指定它，可以确保接收端服务器的中间件能正确识别并处理这个 Excel 文件。

混合上传：文件 + 普通文本（完美合体）。这是实际开发中最常见的场景。比如你在发朋友圈时，**既要上传一张图片（文件），又要写一段文字（文本）**。如果需要在多部分表单中包含非文件数据字段，请使用以下`data=...`参数：

```py
>>> data = {'message': 'Hello, world!'}
>>> with open('report.xls', 'rb') as report_file:
...     files = {'file': report_file}
...     r = httpx.post("https://httpbin.org/post", data=data, files=files)
>>> print(r.text)
{
  ...
  "files": {
    "file": "<... binary content ...>"
  },
  "form": {
    "message": "Hello, world!",
  },
  ...
}
```

**工作原理**：你只需要同时传入 `data`（放文本字典）和 `files`（放文件字典）。

**HTTPX 的幕后黑手**：HTTPX 会非常聪明地将它们打包进同一个 `multipart` 请求体中。它会生成一个特殊的“分隔符（Boundary）”，把文本和文件清晰地隔开，像切香肠一样一段一段地发给服务器。

**返回结果**：如文档输出所示，服务器的 **`"files"` 拿到了文件**，**`"form"` 拿到了文本信息**。



#### 发送 JSON 编码数据

如果只需要简单的键值对数据结构，那么表单编码的数据就可以了。但对于更复杂的数据结构，通常需要使用 JSON 编码。这是目前现代 Web 开发和 RESTful API 交互中**最主流、最核心**的数据提交方式。当你要把一个复杂的 Python 数据结构（比如包含数字、布尔值、嵌套列表或字典的对象）发送给服务器时，你不能用 `data=`，而应该使用 `json=` 参数。

表单编码非常古老，它的底层格式是 `key1=value1&key2=value2`。这种格式有一个致命缺点：**它所有的值在传输时都会被当成“字符串”**。而Json可以做到**数据类型被完美保留下来**。

```py
>>> data = {'integer': 123, 'boolean': True, 'list': ['a', 'b', 'c']}
>>> r = httpx.post("https://httpbin.org/post", json=data)
>>> print(r.text)
{
  ...
  "json": {
    "boolean": true,
    "integer": 123,
    "list": [
      "a",
      "b",
      "c"
    ]
  },
  ...
}
```

##### HTTPX 在幕后帮你做了什么？

当你写下 `r = httpx.post(..., json=data)` 时，HTTPX 在底层偷偷帮你做了两件大事：

1. **自动序列化**：它自动调用了 Python 的 `json.dumps(data)`，把你的 Python 字典翻译成了符合 JSON 规范的字符串。
2. **自动打标签（Content-Type）**：它会自动在请求头（Headers）里加上 `"Content-Type": "application/json"`。这就好比在包裹上贴了一个醒目的标签，告诉接收方服务器：“**注意！我这里面装的是标准的 JSON 货物，请用 JSON 的方式解包！**”

##### `data=` 还是 `json=` ？

在写 `httpx.post()` 时，这俩兄弟长得最像，也是新手最容易写错的地方。我们通过一张对比表彻底分清它们：

| **关键字**     | **底层发送的格式 (Content-Type)**                  | **适合传输的数据**                             | **服务器在哪个字段接收** |
| -------------- | -------------------------------------------------- | ---------------------------------------------- | ------------------------ |
| **`data=...`** | `application/x-www-form-urlencoded` (传统表单格式) | 简单的键值对、传统的网页登录表单（全字符串）   | 服务器的 `"form"` 字段   |
| **`json=...`** | `application/json` (现代化 JSON 格式)              | 复杂的嵌套字典、列表、数字、布尔值等结构化数据 | 服务器的 `"json"` 字段   |

**`params`** = 网址的小尾巴（`?key=value`），**`GET` 和 `POST` 都能用**。

**`data` / `json`** = 信封里面的信件，**只有 `POST` / `PUT` 等能用**，`GET` 不能用。



#### 发送二进制请求数据

`data=`（表单格式）和 `json=`（JSON格式）都是**已经定好规矩的“排版数据”**。而 `content=` 参数，允许你把任何原始的二进制字节流（`bytes`）直接塞进请求体里发出去，没有任何多余的包装。使用`content=...`参数，传递`bytes`类型或生成器以生成`bytes`。

```py
>>> content = b'Hello, world' # 一串纯粹的字节
>>> r = httpx.post("https://httpbin.org/post", content=content)
```

`Content-Type`上传二进制数据时，您可能还需要设置自定义标头。这是使用 `content=` 时**最核心的避坑点**。

- **自动 vs 手动**：
  - 当你用 `data=` 时，HTTPX 自动帮你标记信件为“表单”。
  - 当你用 `json=` 时，HTTPX 自动帮你标记信件为“JSON”。
  - **但是当你用 `content=` 时，HTTPX 保持完全沉默**。它不知道你发的是一首 MP3、一个原始的 XML 文本，还是一段纯文字。它默认不会帮你设置 `Content-Type` 标头。

如果不设置 `Content-Type`，接收端的服务器收到这堆二进制数据时就会一脸懵逼，不知道该怎么解析它。

因此，在实际开发中，使用 `content=` 通常要**搭配自定义的 `headers` 一起使用**：

```py
# 假设你想向服务器发送一段纯明文的 XML 数据
xml_data = b'<xml><message>Hello</message></xml>'

# 必须手动贴上标签，告诉服务器这是 XML
custom_headers = {'Content-Type': 'application/xml'}

r = httpx.post("https://httpbin.org/post", content=xml_data, headers=custom_headers)
```

“传递生成器以生成 bytes” 对应的是 **流式上传（Streaming Uploads）** 场景。 如果你有一个超级大文件（比如 10GB 的视频）要上传，如果直接读到内存里用 `content=file_bytes` 发送，你的电脑内存会瞬间爆满崩溃。

HTTPX 允许你传入一个生成器（Generator），一边读文件，一边发送：

```py
def upload_large_file():
    with open("huge_video.mp4", "rb") as f:
        # 这里使用了 Python 3.8+ 的海象运算符 (:=)
        # 意思是：每次读取 4096 字节（4KB）存入 chunk 变量，如果读到了数据就继续循环
        while chunk := f.read(4096):
            yield chunk  # 核心：吐出这 4KB 数据，然后函数在这里“暂停”

# HTTPX 会维持连接，边读边发，内存永远只占用 4KB
r = httpx.post("https://httpbin.org/post", content=upload_large_file()) 
'''
函数里写了 yield，Python 编译器在编译代码时，就已经给它打上了“特殊标签”。当你调用 upload_large_file() 时，函数体内部的代码其实连第一行都还没开始执行呢，它就直接暂停了！

相反，Python 给你返回了一个特殊的对象——生成器对象（Generator Object）。
'''
```

当执行到 `httpx.post(..., content=upload_large_file())` 时，流程是这样的：

1. **启动**：HTTPX 发现你传给 `content` 的不是死数据，而是一个生成器函数 `upload_large_file()`。它会向这个生成器“索要”第一段数据。（通过检查对象是否具有“迭代属性”（`__iter__`），一眼就识破了它不是死数据，进而用循环调用 `next()` 的方式，主动去唤醒暂停的函数，把数据一点点榨出来。）
2. **第一次循环**：函数进入 `while` 循环，`f.read(4096)` 从硬盘读取了最初的 **4KB** 视频数据，存入 `chunk`。
3. **遇到 `yield`**：程序执行到 `yield chunk`。它把这 4KB 数据递给 HTTPX。**神奇的事情发生了：函数并没有结束，而是像被按了“暂停键”一样，死死记住当前执行到了第 5 行，并且保留了文件打开的状态和所有变量。**
4. **网络传输**：HTTPX 拿到这 4KB 数据，立刻通过网络发给服务器。
5. **再次索要**：HTTPX 发完这 4KB 后，发现网络连接还能继续发，于是再次敲门：“老哥，再给点！”
6. **唤醒与第二次循环**：函数在刚才暂停的第 5 行**被唤醒**，继续往下走，回到 `while` 循环开头。`f.read(4096)` 接着上次的位置，继续从硬盘读入**下一个 4KB** 的数据，再次遇到 `yield`，再次暂停并吐出数据……
7. **终点**：这个过程会循环成千上万次。直到文件读完了，`f.read(4096)` 返回空数据，`while` 循环结束，函数自然退出。HTTPX 知道数据全部传输完毕，关闭网络连接。



#### `data` vs `json` vs `content` 

至此，你已经解锁了 HTTPX 发送请求体的全部三种武器：

| **参数关键字**      | **数据的传输状态**             | **HTTPX 自动设置的 Content-Type**   | **适用场景**                                         |
| ------------------- | ------------------------------ | ----------------------------------- | ---------------------------------------------------- |
| **`data=dict`**     | `key1=value1&key2=value2`      | `application/x-www-form-urlencoded` | 传统网页表单提交、登录                               |
| **`json=dict`**     | `{"key1": "value1"}`           | `application/json`                  | 现代 Web API、微服务通信                             |
| **`content=bytes`** | `value1value2...` (纯原始数据) | **无** (需要你手动指定)             | 发送 XML、发送纯文本、非标准格式数据、流式大文件上传 |

#### `httpx.Response`实例

**是谁实例化的？** 是 **HTTPX 库的内部源码** 自动帮你实例化的，不需要你手动去 `Response()`。

**什么时候实例化的？** 是在**你的网线收到服务器返回的原始数据后，准备把结果交还给你之前**的那一瞬间。

它的内部属性（存数据的地方）

- `r.status_code`：存着一个整数，比如 `200`。
- `r.headers`：存着一个特制的类字典对象，里面是服务器带过来的小纸条。
- `r.content`：存着一串原始的二进制字节（`bytes`）。
- `r.url`：存着这次请求最终指向的 URL 对象。
- `r.encoding`：存着一个字符串（如 `'utf-8'`），记录文本编码。

它的内部方法（动词，能帮你干活的函数）

- `r.json()`：一个方法，点一下，它就去读取 `r.content`，然后用 JSON 工具把它变成 Python 字典。
- `r.raise_for_status()`：一个方法，点一下，它就去检查 `r.status_code` 是不是大于等于 400，是的话就报警。
- `r.read()`：流式读取数据的方法。

在网络世界中，服务器返回的状态码（Status Code）分为两大阵营：

- **成功阵营**：`200~299` 之间的数字（比如 `200 OK`），代表顺利拿到数据。
- **失败阵营**：`400~599` 之间的数字（比如 `404 找不到网页`，`500 服务器内部错误`）。

**HTTPX 的默认脾气**：即使遇到了 404 或 500 这种严重的错误，HTTPX 也觉得“我的网络传输任务完成了，我把对方的报错小纸条带回来了”。所以**它不会让你的 Python 程序崩溃中断**，它依然会老老实实给你返回一个 `<Response [404 Not Found]>` 对象。

如果你希望“**一旦失败，程序就立刻报错中断，别再往下走了**”，你就需要手动调用 `r.raise_for_status()`。

它的逻辑非常简单粗暴：

- 如果状态码是 **200 OK**（成功）：它**什么都不做**，假装无事发生。
- 如果状态码是 **404 或 500**（失败）：它会**立刻掀桌子，抛出一个 `httpx.HTTPStatusError` 异常**，让你的程序停下来。

任何成功的响应代码都会返回`httpx.Response`实例，而不是引发异常。

```py
>>> r.raise_for_status()
```

该方法返回响应实例，允许您直接在代码中使用它。例如：

```py
>>> r = httpx.get('...').raise_for_status()
>>> data = httpx.get('...').raise_for_status().json()
```

#### 流媒体响应

对于大型**下载**，您可能需要使用流式响应，而不是一次性将整个响应体加载到内存中。

前面学过用 `yield` 挤牙膏式**上传**文件，而这里的 `httpx.stream()` 则是用于**下载**时的“挤牙膏”。 当你下载一个 2GB 的电影时，如果直接用 `httpx.get()`，它会试图把 2GB 塞满你的内存。

- `iter_bytes()`：按块读取二进制（适合下载大视频、压缩包）。
- `iter_text()`：按块读取文本字符串。
- `iter_lines()`：**最常用**。按行读取文本。常用于**流式获取大日志文件**，或者现在的 **AI 大模型对话（SSE 协议）**，AI 吐出一个字，你就能打印一个字。
- `iter_raw()`：不解压直接读。即使对方开了 gzip 压缩，也把压缩的原始烂字节交给你。
- ⚠️ **重要避坑**：一旦用了 `stream` 模式，就**不能再调用 `r.text` 或 `r.content`** 了，因为水流过去就没了，内存里没存下整个包裹。
- **技巧**：你可以先开流读取 `Headers`，检查文件多大，太大的话直接挂断不读了（`r.read()` 就是中途反悔，说：“算了，这文件挺小，我全吞下了”）。

您可以流式传输响应的二进制内容：

```py
>>> with httpx.stream("GET", "https://www.example.com") as r:
...     for data in r.iter_bytes():
...         print(data)
```

或者回复的文本……

```py
>>> with httpx.stream("GET", "https://www.example.com") as r:
...     for text in r.iter_text():
...         print(text)
```

或者逐行播放文本……

```py
>>> with httpx.stream("GET", "https://www.example.com") as r:
...     for line in r.iter_lines():
...         print(line)
```

HTTPX 将使用通用换行符，将所有大小写规范化为`\n`.

在某些情况下，您可能希望访问响应中的原始字节，而不进行任何 HTTP 内容解码。在这种情况下，Web 服务器应用的所有内容编码（例如 `<code>` `gzip`、`deflate`` <code> `brotli``、`<code>` 或 ` <code>`）`zstd`都不会自动解码。

```py
>>> with httpx.stream("GET", "https://www.example.com") as r:
...     for chunk in r.iter_raw():
...         print(chunk)
```

如果您以任何上述方式使用流式响应，则 ` `response.content`and``response.text`属性将不可用，并且访问这些属性会引发错误。但是，您也可以使用响应流功能来有条件地加载响应正文：

```py
>>> with httpx.stream("GET", "https://www.example.com") as r:
...     if int(r.headers['Content-Length']) < TOO_LONG:
...         r.read()
...         print(r.text)
```

#### 重定向和历史——网址搬家的幕后追踪

访问 `http://` 网站，服务器会返回 `301` 或 `302` 状态码，意思是：“我已经搬家到 `https://` 了，你自己过去吧”。

- `requests` 默认会自动帮你跳过去，而 **`HTTPX` 默认不会自动跳**（它很严谨，怕你遭遇安全风险）。
- 如果不开启重定向：`r.status_code` 会停在 `301`，你可以通过 **`r.next_request`** 看它本来想让你去哪。
- 开启重定向：必须加上 `follow_redirects=True`。跳转成功后，你可以去 `r.history` 里翻旧账，看看你在到达终点前被中间商倒了多少手。

例如，GitHub 会将所有 HTTP 请求重定向到 HTTPS。

```py
>>> r = httpx.get('http://github.com/')
>>> r.status_code
301
>>> r.history
[]
>>> r.next_request
<Request('GET', 'https://github.com/')>
```

您可以使用以下参数修改默认重定向处理方式`follow_redirects`：

```py
>>> r = httpx.get('http://github.com/', follow_redirects=True)
>>> r.url
URL('https://github.com/')
>>> r.status_code
200
>>> r.history
[<Response [301 Moved Permanently]>]
```

响应的属性`history`可用于检查所有已跟踪的重定向。它包含一个列表，其中列出了所有已跟踪的重定向响应，并按其执行顺序排列。



#### 暂停——预防程序死锁的警报器

HTTPX 默认会为所有网络操作设置合理的超时时间，这意味着如果连接未能正确建立，则应该始终引发错误，而不是无限期地挂起。

网络不活动超时的默认时间为五秒。您可以修改此值以使其更加严格或更宽松：

```py
>>> httpx.get('https://github.com/', timeout=0.001)
```

您也可以完全禁用超时功能……

```py
>>> httpx.get('https://github.com/', timeout=None)
```



#### 验证

HTTPX 支持基本身份验证和摘要式身份验证。

`str`要提供基本身份验证凭据，请将包含纯文本或对象的二元组`bytes`作为`auth`参数传递给请求函数：

```py
>>> httpx.get("https://example.com", auth=("my_user", "password123"))
```

要为摘要式身份验证提供凭据，您需要实例化一个`DigestAuth`对象，并将明文用户名和密码作为参数传递。然后，可以将此对象作为`auth`参数传递给上述请求方法：

```py
>>> auth = httpx.DigestAuth("my_user", "password123")
>>> httpx.get("https://example.com", auth=auth)
<Response [200 OK]>
```



#### 异常处理

如果发生错误，HTTPX 将引发异常。

HTTPX 将网络错误**HTTPError（老祖宗基类）**分成了两个最核心的派系：

 派系 A：**`RequestError`** —— “出师未捷身先死”

- **特点**：请求压根**没发成功**，或者**没收到服务器的有效回复**。
- **场景**：比如断网了、DNS 解析失败域名打不开、或者你刚刚遇到的 **`ReadTimeout`（超时）**。
- **宝贝属性**：你可以通过 `exc.request.url` 拿到当时是去请求哪个倒霉网址时崩掉的。

 派系 B：**`HTTPStatusError`** —— “收到了信，但信里写着拒绝”

- **特点**：网络通畅，服务器也回话了。但你主动调用了 `response.raise_for_status()`，而状态码是 404 或 500。
- **宝贝属性**：不仅能通过 `exc.request` 拿到你的请求，还能通过 `exc.response` 拿到服务器返回的具体报错网页内容（比如对方报错的错误堆栈）。

该类`RequestError`是一个超类，包含了发出 HTTP 请求时发生的任何异常。这些异常包含一个`.request`属性。

```py
try:
    response = httpx.get("https://www.example.com/")
except httpx.RequestError as exc:
    print(f"An error occurred while requesting {exc.request.url!r}.")
```

当响应不是 2xx 成功代码时，`HTTPStatusError`会引发此类异常。这些异常包括 a和 b属性。`response.raise_for_status()``.request``.response`

```py
response = httpx.get("https://www.example.com/")
try:
    response.raise_for_status()
except httpx.HTTPStatusError as exc:
    print(f"Error response {exc.response.status_code} while requesting {exc.request.url!r}.")
```

还有一个`HTTPError`包含这两个类别的基类，可以用来捕获失败的请求或 4xx 和 5xx 响应。

你可以使用这个基类来捕获这两个类别……

```py
try:
    response = httpx.get("https://www.example.com/")
    response.raise_for_status()
except httpx.HTTPError as exc:	
    print(f"Error while requesting {exc.request.url!r}.")
```

或者具体处理每种情况……

```py
try:
    response = httpx.get("https://www.example.com/")
    response.raise_for_status()
except httpx.RequestError as exc:
    print(f"An error occurred while requesting {exc.request.url!r}.")
except httpx.HTTPStatusError as exc:
    print(f"Error response {exc.response.status_code} while requesting {exc.request.url!r}.")
```

## DeepSeek API 指南

### deepseek缓存命中机制

#### 一、 核心前提：缓存落盘与“完整匹配”

在传统机制中，缓存的匹配往往是“能复用多少算多少”（按 Token 逐个匹配）。但由于 Sliding Window Attention 的存在（注意力只关注局部窗口），为了保证计算的准确性和效率，这里引入了“原子化”的概念：

- **缓存前缀单元**：缓存不再是松散的 Token 流，而被切分成了独立、完整的“数据块”。
- **非全即无（All or Nothing）**：后续的请求**必须 100% 完整匹配**某一个已落盘的前缀单元，才能命中这部分缓存。如果只匹配了一半，对不起，这块缓存完全不能用。

#### 二、 三种“落盘”（存入硬盘）的时机

系统通过三种不同的策略，把计算好的缓存切分成“单元”并存入硬盘：

##### 1. 请求结束位置落盘（常规多轮对话场景）

- **机制**：当一次推理结束时（用户输入完、模型回答完），系统会在**输入结束**和**输出结束**这两个节点，分别打包生成缓存前缀单元。
- **效果**：完美契合**标准的多轮对话**（如例一）。因为多轮对话的下一次输入，通常会完整包含上一次的 [输入 + 输出]。

##### 2. 公共前缀检测落盘（“慢半拍”的复用场景）

- **机制**：当系统发现连续几个独立的请求，开头都有一大段一模一样的内容（比如相同的长资料、Prompt 模板），它会把这段公共内容单独打包成一个缓存单元落盘。
- **效果**：如例二所示。第一次和第二次请求虽然开头一样，但由于之前没单独为公共部分落盘，所以第二次**无法命中**（不符合完整匹配）。但系统借此识别出了公共前缀，并完成了落盘，因此第三次请求就能**成功命中**。

##### 3. 按固定 Token 间隔落盘（长文本的保险机制）

- **机制**：如果用户输入了一篇 10 万字的小说，或者模型正在长篇大论地输出，如果非要等到“请求结束”才落盘，一旦中间出错或请求中断，所有的缓存就白计算了。因此，系统会每隔固定的 Token 数（例如每 2048 个 Token）就强制切块落盘一次。
- **效果**：即使是超长文本，后续请求也能命中前面已经切块落盘的“前半部分”缓存。

#### 例一：多轮对话

**第一次请求**

```json
messages: [
    {"role": "system", "content": "你是一位乐于助人的助手"},
    {"role": "user", "content": "中国的首都是哪里？"}
]
```

**第二次请求**

```json
messages: [
    {"role": "system", "content": "你是一位乐于助人的助手"},
    {"role": "user", "content": "中国的首都是哪里？"},
    {"role": "assistant", "content": "中国的首都是北京。"},
    {"role": "user", "content": "美国的首都是哪里？"}
]
```

**解析**

**第一次请求**：`[系统提示词 + 问中国首都]`。请求结束，系统把这一整段打包成**单元①**落盘。

**第二次请求**：`[系统提示词 + 问中国首都 + 答北京 + 问美国首都]`。

**结果**：第二轮的前半部分，**完美且完整地包含了单元①**。因此，单元①对应的缓存**直接命中**，不用重复计算，速度极快。

#### 例二：长文本问答

**第一次请求**

```json
messages: [
    {"role": "system", "content": "你是一位资深的财报分析师..."}
    {"role": "user", "content": "<财报内容>\n\n请总结一下这份财报的关键信息。"}
]
```

**第二次请求**

```json
messages: [
    {"role": "system", "content": "你是一位资深的财报分析师..."}
    {"role": "user", "content": "<财报内容>\n\n请分析一下这份财报的盈利情况。"}
]
```

**第三次请求**

```json
messages: [
    {"role": "system", "content": "你是一位资深的财报分析师..."}
    {"role": "user", "content": "<财报内容>\n\n请分析一下公司收入与支出占比。"}
]
```

**解析:**

**第一次请求**：`[分析师Prompt + 财报内容] + 总结关键信息`。请求结束，整个组合被打包落盘（单元A）。

**第二次请求**：`[分析师Prompt + 财报内容] + 分析盈利`。

- *为什么没命中？* 因为它和单元A的结尾不同（一个是总结信息，一个是分析盈利），无法“完整匹配”单元A。
- *系统的动作*：系统发现两次请求都有 `[分析师Prompt + 财报内容]`，于是把这部分提取出来，单独打包成**单元B**落盘。

**第三次请求**：`[分析师Prompt + 财报内容] + 分析收入支出`。

- *结果*：它的前半部分**完美且完整地匹配了刚刚落盘的单元B**。因此，这部分财报数据的缓存**成功命中**！



### 思考模式

DeepSeek 模型支持思考模式：在输出最终回答之前，模型会先输出一段思维链内容，以提升最终答案的准确性。

#### 思考模式开关与思考强度控制

|                      | 控制参数（OpenAI 格式）                      | 控制参数（Anthropic 格式）                  |
| -------------------- | -------------------------------------------- | ------------------------------------------- |
| 思考模式开关*(1)*    | `{"thinking": {"type": "enabled/disabled"}}` | 同                                          |
| 思考强度控制*(2)(3)* | `{"reasoning_effort": "high/max"}`           | `{"output_config": {"effort": "high/max"}}` |

(1) 默认思考开关为 `enabled`
(2) 思考模式下，对普通请求，默认 effort 为 high；对一些复杂 Agent 类请求（如 Claude Code、OpenCode），effort 自动设置为 `max`
(3) 思考模式下，出于兼容考虑 `low`、`medium` 会映射为 `high`, `xhigh` 会映射为 `max`

在使用 OpenAI SDK 设置 `thinking` 参数时，需要将 `thinking` 参数传入 `extra_body` 中：

```python
response = client.chat.completions.create(
  model="deepseek-v4-pro",
  # ...
  reasoning_effort="high",
  extra_body={"thinking": {"type": "enabled"}}
)
```



#### 输入输出参数

**思考模式不支持 `temperature`、`top_p`、`presence_penalty`、`frequency_penalty` 参数**。请注意，为了兼容已有软件，**设置参数不会报错，但也不会生效**。

在思考模式下，**思维链内容通过 `reasoning_content` 参数返回**，与 `content` 同级。在后续的轮次的拼接中，可以选择性地返回 `reasoning_content` 给 API：

- 在两个 `user` 消息之间，如果模型**未进行工具调用**，则中间 `assistant` 的 `reasoning_content` 无需参与上下文拼接，在后续轮次中将其传入 API 会被忽略。详见[多轮对话拼接](https://api-docs.deepseek.com/zh-cn/guides/thinking_mode#多轮对话拼接)。
- 在两个 `user` 消息之间，如果模型**进行了工具调用**，则中间 `assistant` 的 `reasoning_content` 需参与上下文拼接，在后续所有 user 交互轮次中必须**回传给 API**。详见[工具调用](https://api-docs.deepseek.com/zh-cn/guides/thinking_mode#工具调用)。



#### 多轮对话拼接

在每一轮对话过程中，模型会输出思维链内容（`reasoning_content`）和最终回答（`content`）。如果没有工具调用，则在下一轮对话中，之前轮输出的思维链内容不会被拼接到上下文中，如下图所示：

DeepSeek 模型的思考模式支持工具调用功能。模型在输出最终答案之前，可以进行多轮的思考与工具调用，以提升答案的质量。其调用模式如下图所示：

请注意，区别于思考模式下的未进行工具调用的轮次，进行了工具调用的轮次，在后续所有请求中，必须完整回传 `reasoning_content` 给 API。

若您的代码中未正确回传 `reasoning_content`，API 会返回 400 报错。正确回传方法请您参考下面的样例代码。

##### 样例代码

下面是一个简单的在思考模式下进行工具调用的样例代码：

```python
import os
import json
from openai import OpenAI
from datetime import datetime

# The definition of the tools
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_date",
            "description": "Get the current date",
            "parameters": { "type": "object", "properties": {} },
        }
    },
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get weather of a location, the user should supply the location and date.",
            "parameters": {
                "type": "object",
                "properties": {
                    "location": { "type": "string", "description": "The city name" },
                    "date": { "type": "string", "description": "The date in format YYYY-mm-dd" },
                },
                "required": ["location", "date"]
            },
        }
    },
]

# The mocked version of the tool calls
def get_date_mock():
    return datetime.now().strftime("%Y-%m-%d")

def get_weather_mock(location, date):
    return "Cloudy 7~13°C"

TOOL_CALL_MAP = {
    "get_date": get_date_mock,
    "get_weather": get_weather_mock
}

def run_turn(turn, messages):
    sub_turn = 1
    while True:
        response = client.chat.completions.create(
            model='deepseek-v4-pro',
            messages=messages,
            tools=tools,
            reasoning_effort="high",
            extra_body={ "thinking": { "type": "enabled" } },
        )
        
        messages.append(response.choices[0].message)
        
        reasoning_content = response.choices[0].message.reasoning_content
        content = response.choices[0].message.content
        tool_calls = response.choices[0].message.tool_calls
        print(f"Turn {turn}.{sub_turn}\n{reasoning_content=}\n{content=}\n{tool_calls=}")
        
        # If there is no tool calls, then the model should get a final answer and we need to stop the loop
        if tool_calls is None:
            break
        for tool in tool_calls:
            tool_function = TOOL_CALL_MAP[tool.function.name]
            tool_result = tool_function(**json.loads(tool.function.arguments))
            print(f"tool result for {tool.function.name}: {tool_result}\n")
            messages.append({
                "role": "tool",
                "tool_call_id": tool.id,
                "content": tool_result,
            })
        sub_turn += 1
    print()

client = OpenAI(
    api_key=os.environ.get('DEEPSEEK_API_KEY'),
    base_url=os.environ.get('DEEPSEEK_BASE_URL'),
)

# The user starts a question
turn = 1
messages = [{
    "role": "user",
    "content": "How's the weather in Hangzhou Tomorrow"
}]
run_turn(turn, messages)

# The user starts a new question
turn = 2
messages.append({
    "role": "user",
    "content": "How's the weather in Guangzhou Tomorrow"
})
run_turn(turn, messages)
```

在 Turn 1 的每个子请求中，都携带了该 Turn 下产生的 `reasoning_content` 给 API，从而让模型继续之前的思考。`response.choices[0].message` 携带了 `assistant` 消息的所有必要字段，包括 `content`、`reasoning_content`、`tool_calls`。简单起见，可以直接用如下代码将消息 append 到 messages 结尾：

```text
messages.append(response.choices[0].message)
```

这行代码等价于：

```text
messages.append({
    'role': 'assistant',
    'content': response.choices[0].message.content,
    'reasoning_content': response.choices[0].message.reasoning_content,
    'tool_calls': response.choices[0].message.tool_calls,
})
```

### 多轮对话

DeepSeek `/chat/completions` API 是一个“无状态” API，即服务端不记录用户请求的上下文，用户在每次请求时，**需将之前所有对话历史拼接好后**，传递给对话 API。

下面的代码以 Python 语言，展示了如何进行上下文拼接，以实现多轮对话。

```python
from openai import OpenAI
client = OpenAI(api_key="<DeepSeek API Key>", base_url="https://api.deepseek.com")

# Round 1
messages = [{"role": "user", "content": "What's the highest mountain in the world?"}]
response = client.chat.completions.create(
    model="deepseek-v4-pro",
    messages=messages
)

messages.append(response.choices[0].message)
print(f"Messages Round 1: {messages}")

# Round 2
messages.append({"role": "user", "content": "What is the second?"})
response = client.chat.completions.create(
    model="deepseek-v4-pro",
    messages=messages
)

messages.append(response.choices[0].message)
print(f"Messages Round 2: {messages}")
```

在**第一轮**请求时，传递给 API 的 `messages` 为：

```json
[
    {"role": "user", "content": "What's the highest mountain in the world?"}
]
```

在**第二轮**请求时：

1. 要将第一轮中模型的输出添加到 `messages` 末尾
2. 将新的提问添加到 `messages` 末尾

最终传递给 API 的 `messages` 为：

```json
[
    {"role": "user", "content": "What's the highest mountain in the world?"},
    {"role": "assistant", "content": "The highest mountain in the world is Mount Everest."}, 
    {"role": "user", "content": "What is the second?"}
]
```



### 对话前缀续写（Beta）

对话前缀续写沿用 [Chat Completion API](https://api-docs.deepseek.com/zh-cn/api/create-chat-completion)，用户提供 assistant 开头的消息，来让模型补全其余的消息。

#### 注意事项

1. 使用对话前缀续写时，**用户需确保 `messages` 列表里最后一条消息的 `role` 为 `assistant`**，**并设置最后一条消息的 `prefix` 参数为 `True`。**
2. 用户需要设置 `base_url="https://api.deepseek.com/beta"` 来开启 Beta 功能。

#### 样例代码

下面给出了对话前缀续写的完整 Python 代码样例。在这个例子中，我们设置 `assistant` 开头的消息为 `"```python\n"` 来强制模型输出 python 代码，并设置 `stop` 参数为 `['```']` 来避免模型的额外解释。

```python
from openai import OpenAI

client = OpenAI(
    api_key="<your api key>",
    base_url="https://api.deepseek.com/beta", # 多个beta
)

messages = [
    {"role": "user", "content": "Please write quick sort code"},
    {"role": "assistant", "content": "```python\n", "prefix": True}
]
response = client.chat.completions.create(
    model="deepseek-v4-pro",
    messages=messages,
    stop=["```"],
)
print(response.choices[0].message.content)
```



### FIM 补全（Beta）

在 [FIM (Fill In the Middle) 补全](https://api-docs.deepseek.com/zh-cn/api/create-completion)中，**用户可以提供前缀和后缀**（可选），**模型来补全中间的内容**。FIM 常用于**内容续写**、代码补全等场景。

#### 注意事项

1. 模型的最大补全长度为 4K。
2. 用户需要设置 `base_url="https://api.deepseek.com/beta"` 来开启 Beta 功能。

#### 样例代码

下面给出了 FIM 补全的完整 Python 代码样例。在这个例子中，我们给出了计算斐波那契数列函数的开头和结尾，来让模型补全中间的内容。

```python
from openai import OpenAI

client = OpenAI(
    api_key="<your api key>",
    base_url="https://api.deepseek.com/beta",
)

response = client.completions.create(
    model="deepseek-v4-pro",
    prompt="def fib(a):",
    suffix="    return fib(a-1) + fib(a-2)",
    max_tokens=128
)
print(response.choices[0].text)
```



### JSON Output

在很多场景下，用户需要让模型严格按照 JSON 格式来输出，以实现输出的结构化，便于后续逻辑进行解析。

DeepSeek 提供了 JSON Output 功能，来确保模型输出合法的 JSON 字符串。

#### 注意事项

1. 设置 `response_format` 参数为 `{'type': 'json_object'}`。
2. 用户传入的 system 或 user prompt 中必须含有 `json` 字样，并给出希望模型输出的 JSON 格式的样例，以指导模型来输出合法 JSON。
3. 需要合理设置 `max_tokens` 参数，防止 JSON 字符串被中途截断。
4. **在使用 JSON Output 功能时，API 有概率会返回空的 content。我们正在积极优化该问题，您可以尝试修改 prompt 以缓解此类问题。**

#### 样例代码

这里展示了使用 JSON Output 功能的完整 Python 代码：

```python
import json
from openai import OpenAI

client = OpenAI(
    api_key="<your api key>",
    base_url="https://api.deepseek.com",
)

# 系统提示词，用于设定场景、规则。举例可以规范化模型的输出
system_prompt = """
The user will provide some exam text. Please parse the "question" and "answer" and output them in JSON format. 

EXAMPLE INPUT: 
Which is the highest mountain in the world? Mount Everest.

EXAMPLE JSON OUTPUT:
{
    "question": "Which is the highest mountain in the world?",
    "answer": "Mount Everest"
}
"""

user_prompt = "Which is the longest river in the world? The Nile River."

messages = [{"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt}]

response = client.chat.completions.create(
    model="deepseek-v4-pro",
    messages=messages,
    response_format={
        'type': 'json_object'
    }
)

print(json.loads(response.choices[0].message.content))
```

模型将会输出：

```json
{
    "question": "Which is the longest river in the world?",
    "answer": "The Nile River"
}
```



### Tool Calls

#### 非思考模式——tool函数由用户提供

这里以获取用户当前位置的天气信息为例，展示了使用 Tool Calls 的完整 Python 代码。

Tool Calls 的具体 API 格式请参考[对话补全](https://api-docs.deepseek.com/zh-cn/api/create-chat-completion/)文档。

```python
from openai import OpenAI

def send_messages(messages):
    response = client.chat.completions.create(
        model="deepseek-v4-pro",
        messages=messages,
        tools=tools
    )
    return response.choices[0].message

client = OpenAI(
    api_key="<your api key>",
    base_url="https://api.deepseek.com",
)

tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get weather of a location, the user should supply a location first.",
            "parameters": {
                "type": "object",
                "properties": {
                    "location": {
                        "type": "string",
                        "description": "The city and state, e.g. San Francisco, CA",
                    }
                },
                "required": ["location"]
            },
        }
    },
]

messages = [{"role": "user", "content": "How's the weather in Hangzhou, Zhejiang?"}]
message = send_messages(messages)
print(f"User>\t {messages[0]['content']}")

tool = message.tool_calls[0]
messages.append(message)

# 添加工具调用消息
messages.append({"role": "tool", "tool_call_id": tool.id, "content": "24℃"})
message = send_messages(messages) # 给模型发送带有用户、工具的所有。输出包括模型根据tool的结果发送的内容
print(f"Model>\t {message.content}")
```



这个例子的执行流程如下：

1. 用户：询问现在的天气
2. 模型：返回 function `get_weather({location: 'Hangzhou'})`
3. 用户：调用 function `get_weather({location: 'Hangzhou'})`，并传给模型。
4. 模型：返回自然语言，"The current temperature in Hangzhou is 24°C."

注：上述代码中 **`get_weather` 函数功能需由用户提供，模型本身不执行具体函数。**

#### 思考模式

从 DeepSeek-V3.2 开始，API 支持了思考模式下的工具调用能力，详见[思考模式](https://api-docs.deepseek.com/zh-cn/guides/thinking_mode#工具调用)。

------

#### `strict` 模式（Beta）

在 `strict` 模式下，模型在输出 Function 调用时会**严格遵循 Function 的 JSON Schema 的格式要求**，以确保模型输出的 Function **符合用户的定义**。在思考与非思考模式下的工具调用，均可使用 `strict` 模式。

要使用 `strict` 模式，需要：

1. 用户需要设置 `base_url="https://api.deepseek.com/beta"` 来开启 Beta 功能
2. 在传入的 `tools` 列表中，所有 `function` 均需设置 `strict` 属性为 `true`
3. 服务端会对用户传入的 Function 的 JSON Schema 进行校验，如不符合规范，或遇到服务端不支持的 JSON Schema 类型，将返回错误信息

以下是 `strict` 模式下 tool 的定义样例：

```json
{
    "type": "function",
    "function": {
        "name": "get_weather",
        "strict": true,
        "description": "Get weather of a location, the user should supply a location first.",
        "parameters": {
            "type": "object",
            "properties": {
                "location": {
                    "type": "string",
                    "description": "The city and state, e.g. San Francisco, CA",
                }
            },
            "required": ["location"],
            "additionalProperties": false
        }
    }
}
```



------

#### `strict` 模式支持的 JSON Schema 类型

- object
- string
- number
- integer
- boolean
- array
- enum
- anyOf

------

##### object 类型

object 定义一个包含**键值对**的深层结构，其中 properties 定义了对象中每个键（属性）的 schema。**每个 `object` 的所有属性均需设置为 `required`，且 `object` 中 `additionalProperties` 属性必须为 `false`**。

示例：

```json
{
    "type": "object",
    "properties": {
        "name": { "type": "string" },
        "age": { "type": "integer" }
    },
    "required": ["name", "age"],
    "additionalProperties": false
}
```



------

##### string 类型

- 支持的参数：
  - pattern：使用正则表达式来约束字符串的格式
  - format：使用预定义的常见格式进行校验，目前支持：
    - email：电子邮件地址
    - hostname：主机名
    - ipv4：IPv4 地址
    - ipv6：IPv6 地址
    - uuid：uuid
- 不支持的参数
  - minLength
  - maxLength

示例：

```json
{
    "type": "object",
    "properties": {
        "user_email": {
            "type": "string",
            "description": "The user's email address",
            "format": "email" 
        },
        "zip_code": {
            "type": "string",
            "description": "Six digit postal code",
            "pattern": "^\\d{6}$"
        }
    }
}
```



------

##### number/integer 类型

- 支持的参数
  - const：固定数字为常数
  - default：数字的默认值
  - minimum：最小值
  - maximum：最大值
  - exclusiveMinimum：不小于
  - exclusiveMaximum：不大于
  - multipleOf：数字输出为这个值的倍数

示例：

```text
{
    "type": "object",
    "properties": {
        "score": {
            "type": "integer",
            "description": "A number from 1-5, which represents your rating, the higher, the better",
            "minimum": 1,
            "maximum": 5
        }
    },
    "required": ["score"],
    "additionalProperties": false
}
```



------

array 类型

- 不支持的参数
  - minItems
  - maxItems

示例：

```json
{
    "type": "object",
    "properties": {
        "keywords": {
            "type": "array",
            "description": "Five keywords of the article, sorted by importance",
            "items": {
                "type": "string",
                "description": "A concise and accurate keyword or phrase."
            }
        }
    },
    "required": ["keywords"],
    "additionalProperties": false
}
```



------

##### enum

enum 可以确保输出是预期的几个选项之一，例如在订单状态的场景下，只能是有限几个状态之一。

样例：

```text
{
    "type": "object",
    "properties": {
        "order_status": {
            "type": "string",
            "description": "Ordering status",
            "enum": ["pending", "processing", "shipped", "cancelled"]
        }
    }
}
```



------

##### anyOf

匹配所提供的多个 schema 中的任意一个，可以处理可能具有多种有效格式的字段，例如用户的账户可能是邮箱或者手机号中的一个：

```json
{
    "type": "object",
    "properties": {
    "account": {
        "anyOf": [
            { "type": "string", "format": "email", "description": "可以是电子邮件地址" },
            { "type": "string", "pattern": "^\\d{11}$", "description": "或11位手机号码" }
        ]
    }
  }
}
```



------

##### $ref 和 $def

可以使用 $def 定义模块，再用 $ref 引用以减少模式的重复和模块化，此外还可以单独使用 $ref 定义递归结构。

```json
{
    "type": "object",
    "properties": {
        "report_date": {
            "type": "string",
            "description": "The date when the report was published"
        },
        "authors": {
            "type": "array",
            "description": "The authors of the report",
            "items": {
                "$ref": "#/$def/author"
            }
        }
    },
    "required": ["report_date", "authors"],
    "additionalProperties": false,
    "$def": {
        "author": {
            "type": "object",
            "properties": {
                "name": {
                    "type": "string",
                    "description": "author's name"
                },
                "institution": {
                    "type": "string",
                    "description": "author's institution"
                },
                "email": {
                    "type": "string",
                    "format": "email",
                    "description": "author's email"
                }
            },
            "additionalProperties": false,
            "required": ["name", "institution", "email"]
        }
    }
}
```
